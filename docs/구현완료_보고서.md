## 1. 요약 (Executive Summary)

GeoAerospace는 "위성 궤도의 구형(球) 시각화"를 북극성 요구로 삼는 대화형 지오-플랫폼이다. 현재 상태를 한 문장으로 요약하면 다음과 같다 — **시각화·데이터 수집 계층은 상당한 완성도에 도달했고, 지능 계층(에이전트·RAG·VLM)은 동작하되 방향이 지정한 도구 스택과는 아직 간극이 있다.**

완성된 것의 핵심은 세 가지다. 첫째, **위성 궤도의 구형 시각화**를 서로 다른 문제를 푸는 세 층위(2D 글로브 · 관성계 3D 뷰 · 지도 위 3D 레이어)로 나누어 구현했고, 이는 참조 저장소(NASA Worldview·OSIRIS·WorldWideView)가 모두 2D에 머문 영역이라 이 프로젝트의 실질적 차별점이다. 둘째, **실시간 항공 트래킹**을 ADS-B 다중 폴백과 추측항법(dead-reckoning) 보간으로 구현해 한 화면에 약 2,000대를 부드럽게 렌더한다. 셋째, 궤도·좌표·산불·한반도 관측(GK2A)에 이르는 **데이터 파이프라인 전반을 실측으로 검증**했으며, 자체 구현한 수학(좌표 프레임 변환·LCC 투영·기동 감지)을 독립 기준 구현과 수치 대조해 오차를 정량화했다.

동시에 이 보고서는 **정직한 한계**도 함께 기록한다. 지정 스택인 LangChain deepagents(Python 에이전트 계층)는 아직 도입되지 않았고 현재는 TypeScript 규칙 기반이며, RAG 코퍼스는 소규모(14청크)이고, VLM은 파이프라인은 정상이나 범용 모델이라 원격탐사 해석에 실패한다. 이 간극들은 §8에 방향 대비로 명시한다.

## 2. 프로젝트 정체성과 기술 스택

플랫폼의 목표는 사용자가 자연어로 명령하면 위성 궤도·항공기·위성영상 데이터를 검색·해석해 **3D 지구본 위에 시각화**하는 것이다. 설계 철학의 핵심 원칙은 "**VLM이 궤도 수치를 해석한다는 것은 범주 오류(category error)**"라는 분리 원칙이다 — 궤도 계산은 결정론적 물리 엔진(SGP4)이, 영상 해석은 VLM이, 지식 검색은 RAG가 담당하고, LLM은 이 특화 컴포넌트들을 조율하는 에이전트로만 동작한다.

## 4. 요구사항 ① — 위성 궤도의 구형(球) 시각화 [완료]

궤도를 구에 올리는 방법은 하나가 아니다. 이 프로젝트는 **세 층위로 나누어** 구현했고, 각 층위가 푸는 문제가 다르다. 이 다층 구조가 완성된 부분 중 기술적으로 가장 밀도가 높다.

### 4.1 세 가지 구형 표현

| 층위 | 기술 | 좌표계 | 푸는 문제 |
|---|---|---|---|
| **A. MapLibre 글로브** | MapLibre GL JS v5 `projection: globe` | 지구고정(경위도) | 2D 지도를 그대로 구에 감아 지상궤적·항공기·산불을 구 위에서 본다 |
| **B. Three.js 관성계 뷰** | 커스텀 `SpaceView`(WebGL) | ECI(관성계) | 궤도 타원을 실축척으로 공간에 띄운다 |
| **C. MapLibre 위 3D 레이어** | Three.js custom layer(`orbital-3d`) | 지구고정 | 2D 지도 위에 위성 본체·센서 콘을 3D로 얹는다 |

세 층위가 모두 필요한 이유는 물리적으로 명확하다. 지구고정 좌표에 궤도를 그리면 지구 자전 때문에 궤도가 매 순간 휘어 보이고(지상궤적이 사인 곡선이 되는 이유), 반대로 관성계에 그리면 궤도는 고정된 타원이고 지구가 그 안에서 자전한다 — 후자가 물리적으로 옳은 그림이다. 두 그림이 모두 필요하므로 뷰를 나눴다.

### 4.2 B층 — 관성계 구형 뷰 (핵심 구현)

가장 핵심적인 산출물이며, `components/SpaceView.tsx`에 구현되어 있다(셰이더 포함, 단일 파일 약 900행 규모).

**좌표 규약과 자전 처리.** 씬 단위는 `1 scene unit = 1000 km`, 지구 반경 `R = 6.371 unit`이며, ECI(Z-up, km)를 Three.js(Y-up, unit)로 `(x, z, −y) × 0.001`로 사상한다. 코드에서 `earth.rotation.y = satellite.gstime(now)` 한 줄이 GMST 자전을 담당하고 궤도 링은 ECI 좌표를 그대로 두어 회전시키지 않는다 — 이 한 줄이 "관성계 뷰"의 전부다. 궤도가 공간에 못 박혀 있고 지구가 그 안에서 자전한다.

**궤도 타원 생성.** 한 주기를 256등분해 SGP4로 전파하고 ECI 좌표를 그대로 이어 `THREE.Line`으로 그린다. 지구고정 변환을 거치지 않으므로 타원이 닫힌다. (2D 글로브용 링·지상궤적은 `lib/orbit.ts`의 `computeOrbit()`가 한 주기를 180등분 샘플링하고, 자오선 ±180° 교차 시 지상궤적 세그먼트를 분할해 래핑 아티팩트를 방지한다 — 코드에서 확인됨.)

**지구 표면 — 셰이더 5겹 합성.** 단순 텍스처가 아니라 프래그먼트 셰이더에서 다섯 겹을 합성한다: (1) 당일 트루컬러 → 결손 시 전날 → 결손 시 정적 BlueMarble의 3단 알베도 폴백, (2) 야간 도시광(밤면), (3) 태양각 기반 터미네이터 + 박명 띠, (4) GK2A 관측 격자 오버레이(한반도). 등장방형(equirectangular) UV라 `lon = u×360−180`, `lat = v×180−90`으로 경위도를 역산할 수 있어 임의 영역 오버레이를 셰이더에서 직접 합성한다.

**대기.** 지구보다 3.5% 큰 구를 `BackSide` + 가산 블렌딩으로 그려 림 글로우를 만들고, Rayleigh(청)·Mie(주황)를 태양각으로 가중한다. **물리 산란 적분이 아닌 경험적 근사**임을 문서와 코드 주석 모두에 정직하게 명시한다.

### 4.3 위성 본체 — 스케일 문제와 절차적 모델

`lib/three/satelliteModel.ts`에 구현. 실축척으로는 위성이 보이지 않는다는 근본 문제를 정면으로 다룬다: ISS 실제 크기 109 m = 0.000109 unit은 서브픽셀이라 화면에 나타나지 않으며, 기존 구 마커(0.12 unit = 120 km)조차 이미 1,100배 과장된 값이었다. 이 배율에서 위성은 화면상 10~30 px이라 고밀도 glTF의 디테일은 대부분 버려진다. 그래서 **절차적 모델**(ISS 트러스·모듈·태양전지판 / Hubble 경통 / 범용 버스)로 실루엣만 만들고 GLTFLoader 연결부를 남겨 실제 모델로 교체 가능하게 했다.

코드 대조 결과 다음 세 요소가 문서 서술대로 구현되어 있음을 확인했다.

- **LVLH 자세 정렬** (`applyNadirAttitude`): `z = −r̂`(천저), `y = −ĥ`(h = r×v), `x = y×z`(≈비행 방향)로 기저를 구성하고, 수치 오차를 `y = z×x` 재직교화로 보정한다. 축퇴(속도 0)에서는 자세를 유지한다.
- **태양전지판 태양 추적** (`trackSun`): 월드 태양 벡터를 모델 좌표계로 옮겨 X–Z 평면 투영각(`atan2(sun.x, sun.z)`)으로 피벗을 Y축 회전한다.
- **마커 LOD** (`markerSizing`): 모델만 두면 식(eclipse) 중인 10 px 모델이 어두운 지구에 묻혀 추적 기능을 잃으므로, 화면 크기 고정 글로우 마커(약 13 px)를 병행하고 모델이 26~60 px로 커지면 마커를 페이드아웃한다. 마커는 식 중에도 항상 밝게 유지해 추적성을 보존한다.

### 4.4 위성 시점(POV)과 관측 격자 재투영

**POV 뷰.** 위성을 클릭하면 그 위성이 보는 화면이 성립한다(정지궤도 GK2A에서 지구 각반경 8.7°가 45° 화각에 담김). 구현 시 두 함정(자기 센서 콘 내부를 보게 됨 / 자기 본체가 시야를 가림)을 잡아, POV에서는 자기 모델과 자기 콘을 숨긴다.

**LCC 재투영** (`lib/lcc.ts` · `lib/gk2aRaster.ts`). 기상청 GK2A 격자는 Lambert Conformal Conic이고 구는 등장방형 UV라, 그대로 얹으면 한반도가 엉뚱한 곳에 찍힌다. **역방향 매핑**으로 리샘플링해 — 출력 픽셀마다 경위도를 구해 격자에서 값을 "당겨온다"(inverse) — 구멍·겹침을 피한다. 검증에서 서울·부산·제주·강릉 모두 원래 경위도에 **0.1° 이내**로 안착했다.

## 5. 요구사항 ② — 항공 트래킹 [완료]

### 5.1 데이터 소스와 실시간성

전용 항공 라이브러리를 쓰지 않고 ADS-B 공개 API를 직접 호출한다 — `api.adsb.lol`(1차) → `api.airplanes.live`(폴백). 필요한 값은 좌표·고도·방위·속도 네 개뿐이라 라이브러리를 끼우면 의존성과 폴백 제어 부담만 늘기 때문이다. `lib/server/fetchAircraft.ts`가 7개 허브 지역(프랑크푸르트·런던·뉴욕·LA·인천·도쿄·두바이, 반경 250 nm) 팬아웃, `Promise.allSettled`, single-flight, 10초 캐시, 429 시 15분 쿨다운, ICAO 콜사인 정규식 분류(상용/개인/제트/군용)를 담당한다. 호스트는 `lib/server/safeFetch.ts`의 SSRF 가드(allowlist + https 강제 + `redirect:'error'`)로 관리한다.

### 5.2 추측항법(dead-reckoning) 보간

ADS-B 폴링은 약 12초 간격인데 항공기는 250 m/s로 움직여, 폴링 간격 동안 3 km를 이동하므로 마커가 끊긴다. `lib/aircraft.ts`의 `deadReckon()`은 마지막 스냅숏의 위치·방위·대지속도로 매 프레임 대권(great-circle) 전진을 계산한다. 코드 대조 결과 `destination(lon, lat, distKm, bearing)`이 구면 삼각법(하버사인 계열)으로 정확히 구현되어 있고, 30 fps로 부드럽게 움직이다가 다음 폴링에서 실측으로 보정된다.

### 5.3 렌더링 — 수천 대를 60 fps로

deck.gl `IconLayer`(MapboxOverlay interleaved)로 GPU 인스턴싱해 draw call 한 번에 수천 마커를 그린다. `getAngle: −track`으로 기수 방위 회전, 카테고리별 색상(코드의 `AC_COLOR` 테이블 확인), `useRef` + 수동 리렌더(수천 엔티티를 React state에 넣지 않음), `document.hidden` 가드(백그라운드 탭 폴링 중단)를 적용한다. 아이콘은 외부 자산 없이 런타임에 캔버스로 생성한다(`makePlaneIcon` — 코드 확인). **실측: 세션에 따라 약 1,686~2,298대 동시 표시**, 실제 콜사인(UPS4·EZY8590·JAF45T 등)·고도·속도·heading 확인됨.

### 5.4 위성과 항공의 통합

이 프로젝트는 위성 궤도와 항공 항적을 같은 플랫폼에서 다룬다. 다만 고도 스케일이 2,500배 차이(위성 400~35,800 km vs 항공 0~14 km)라 같은 3D 씬에 실축척으로 두면 항공기가 지표에 붙어 보인다. 현재는 **2D 글로브(A층)에서 항공기를, 관성계 뷰(B층)에서 위성을** 주로 다루는 것으로 정리했고, 완전한 동시 3D 통합은 미해결 과제로 남겼다(§9).

## 6. 데이터 계층 — 실측 검증 완료

기준 문서와 상세 검증 문서(`구현기록_기술근거와_검증.md`)가 기록한 데이터 파이프라인은 모두 **실제 네트워크 호출과 기준 구현 대조로 검증**되었다.

**궤도 원소 수집.** Space-Track REST를 1차로 하여 세션 쿠키 90분 재사용 + 전 위성 1회 배치 쿼리로 레이트리밋(분당 30/시간당 300)을 보호한다. CelesTrak은 이 네트워크에서 차단(http=000)되어 SatNOGS 폴백으로 동작함을 확인했다. 소스 표기를 정직화해 데모/스테일이 섞이면 `LIVE`가 아니라 `PARTIAL`로 표기한다.

**정밀 궤도와 오차 정량화.** ISS는 NASA OEM(±8시간 창, 8점 Lagrange 보간 + J2000→TEME 변환)을 렌더 좌표로 직접 사용한다. SGP4의 실제 오차를 OEM 대조로 측정한 결과 epoch 시점에 이미 0.681 km 어긋나 있고 오차는 거의 전부 along-track이며, 증가율은 비선형이라 단일 수치로 말할 수 없다(0~72h 0.02 km/일 → 0~240h 2.58 km/일). 이 실측이 원안의 "수치적분으로 sub-100 m" 전제를 반증했고, 그 결과 불필요한 Python 마이크로서비스를 만들지 않는 판단으로 이어졌다.

**기동 감지.** `lib/maneuvers.ts`가 반장축 계단 상승을 MAD 기반 강건 임계값으로 탐지한다. 실제 Space-Track 이력 272건(70일)에서 ISS 재부스트 2건(2026-06-10 +1.55 km, 2026-07-01 +1.83 km)을 항력 잡음 대비 20~40배 크기·반대 부호로 검출했다.

**가시성·통과 예측.** `lib/passes.ts`가 지평좌표 변환 + 조밀탐색·이분법 AOS/LOS 정밀화를 수행하며, 지상국별 `min_horizon`(실측 중앙값 10°, 범위 0~70°)을 존중한다. Python 독립 구현과 6/6 완전 일치로 교차검증했다.

**GK2A(천리안위성 2A호) 한반도 관측.** LCC 좌표 변환(가이드 기준값 대비 오차 1e-6° ≈ 0.1 m), Count→물리량 변환(변환표 계수 내장, 소수점 8자리 일치), 2분 간격 시계열(연속 프레임 90%+ 변화로 반복 데이터가 아님을 확인)을 구현·검증했다. 다만 관측 지연이 6시간~2일이라 "실시간"이 아니라 "시간 해상도"를 제공한다는 점을 명확히 한다.

**산불 3계층.** FIRMS 활성 화재(키 없는 Open Data CSV + 키 있으면 Area API), GIBS 맥락영상, VLM 해석의 3계층. FIRMS 필터는 원본 CSV 16만 행 직접 필터와 완전 일치로 교차검증했다("캘리포니아 FRP 100 MW 이상 5건, 최대 231.1 MW").

## 7. 지능 계층 — 동작하되 방향과 간극

### 7.1 에이전트 (현재: TypeScript 규칙 기반)

`lib/agent.ts`의 `resolveIntent()`가 정규식으로 의도를 판정하고 도구를 실행한다. 코드 대조 결과 현재 도구 6종(`fly_to_place`·`select_satellite`·`toggle_layer`·`filter_fires`·`add_layer`·`analyze_image`)과 **도구 연쇄**가 구현되어 있으며, 산불 질의 하나가 `filter_fires → add_layer → analyze_image` 3계층(fire-analyst)으로 전개된다.

규칙 기반을 택한 근거도 코드와 문서 모두에서 확인된다 — 로컬 소형 모델(Qwen3-8B)의 도구 선택 변동성이 커서("서울로 이동"을 `toggle_layer(terrain)`으로 오분류하는 등) 지도 명령은 결정론적으로 처리하고, 좌표는 모델이 아니라 시스템이 지오코딩(도시 테이블 + Nominatim)으로 만들어 환각을 차단한다.

### 7.2 RAG (현재: 소규모 사실 코퍼스)

bge-m3(Ollama) 임베딩 + 코사인 유사도·어휘 부스트 하이브리드 검색 + Qwen3-8B 종합으로 궤도역학 Q&A를 제공한다. 코퍼스는 `lib/knowledge.ts`의 손으로 쓴 14청크로, 방향이 지향하는 "궤도 데이터를 많이 끌고 오는 RAG"가 아니라 궤도역학 용어 Q&A 수준이다.

### 7.3 VLM (현재: 연결됨, 그러나 신뢰 불가 — 정직한 기록)

`app/api/analyze-image/route.ts`가 GIBS WMS로 관심영역을 잘라(768 px) qwen2.5vl:7b(Ollama)에 전달한다. 코드 대조 결과 **파이프라인은 정상**이고 FIRMS 수치를 프롬프트에 함께 넣는 환각 억제 그라운딩까지 구현되어 있으나, 범용 모델이라 7월 캘리포니아 영상을 "눈 덮인 산악"으로 해석하고 화재를 놓쳤다. 원인은 범용 VLM에 원격탐사 도메인 그라운딩이 없다는 것이며, 이는 제안서가 GeoChat을 지목한 이유와 정확히 일치한다. **교체 지점은 환경변수 한 곳**(`VLM_MODEL`/`OLLAMA_URL`)으로 설계되어 있어(코드 확인), 원격탐사 특화 모델 확보 시 최소 변경으로 교체 가능하다.

## 8. 방향(문서) 대비 구현(코드) 대조

기준 문서가 정직하게 짚은 세 간극을 코드 대조 결과와 함께 정리한다. 시각화·데이터 계층은 방향과 어긋남이 없고, 지능 계층에 간극이 집중된다.

| 방향 | 현재 실태 | 간극 |
|---|---|---|
| MapLibre 웹 환경 | ✅ MapLibre v5 + deck.gl + Three.js | 없음 |
| 구형 시각화 | ✅ 3층위 구현(§4) | 없음 |
| 항공 트래킹 | ✅ ADS-B 실시간(§5) | 없음 |
| 데이터 파이프라인 | ✅ 9소스 실측 검증(§6) | 없음 |
| **LangChain deepagents** | ❌ TypeScript 규칙 기반 + Ollama 직접 호출 | **큼** — Python 에이전트 계층 부재 |
| **궤도 데이터 RAG** | ⚠️ 코퍼스 14청크(손으로 씀) | **큼** — 대규모 궤도 RAG 아님 |
| **VLM 궤도 해석** | ⚠️ 연결되나 범용 모델이라 해석 실패 | **큼** — GeoChat 미도입 |

기준 문서가 제시한 전환 경로(deepagents는 `ollama:` 접두사로 현재 로컬 모델 자산을 그대로 재사용 가능, GeoChat은 환경변수 한 곳 교체, RAG는 기동 감지 결과 자동 서술화부터 확장)는 모두 기존 구조 위에 얹는 방식이라 되돌리기 쉬운 순서(GeoChat → RAG 확장 → deepagents 전환)로 권장되어 있다.

## 9. 정확도·검증 방법론

이 프로젝트의 특징은 자체 구현한 수학을 **독립 기준 구현과 수치 대조**하고, 검증 도구 자체의 결함까지 잡아냈다는 점이다.

| 대상 | 기준 | 잔차 |
|---|---|---|
| SGP4 전파 | Vallado sgp4-ver 골든 벡터(catalog 00005) | < 10 m / 1 mm·s |
| TEME↔J2000 | Skyfield(IAU 전항 구현) | 0.8 m |
| LCC 변환 | 기상청 가이드 명시 기준값 | 1e-6° (0.1 m) |
| Count→물리량 | GK2A 변환표 룩업 행 | 소수점 8자리 |
| 지상국 가시성 | Python 독립 구현 | 6/6 완전 일치 |
| FIRMS 필터 | 원본 CSV 직접 필터 | 완전 일치 |

검증 도구를 의심한 사례도 기록되어 있다 — 2점 Hermite 보간의 고유 오차 94.65 m가 측정값을 오염시킨 것을 8점 Lagrange로 교정했고, 캡처 훅이 렌더 루프를 복제해 거짓을 보여준 것을 `updateScene()` 통합으로 해결했다. 서버 정확도 CI(골든값 오차 초과 시 배포 차단)가 `npm test`에 포함된다.

## 12. 결론

기준 문서(`프로젝트구성_구형시각화와_항공트래킹`)가 정의한 두 요구사항 — **위성 궤도의 구형 시각화**와 **항공 트래킹** — 은 모두 구현이 완료되었고, 소스 코드 대조로 문서 서술과 구현이 일치함을 확인했다. 구형 시각화는 세 층위(2D 글로브·관성계 3D·지도 위 3D 레이어)로 나누어 참조 저장소들이 하지 못한 영역을 구현했고, 항공 트래킹은 다중 폴백과 추측항법으로 약 2,000대를 실시간 렌더한다. 그 아래의 데이터 계층은 9개 소스를 실측하고 자체 수학을 독립 기준 구현과 대조해 오차를 정량화했으며, 학술 인용까지 서지 검증을 마쳤다.

남은 과제는 명확하다 — 지능 계층을 방향이 지정한 스택(deepagents · GeoChat · 대규모 궤도 RAG)으로 전환하는 것이다. 이 전환들은 모두 기존 구조 위에 얹는 방식으로 설계되어 있어(로컬 모델 자산 재사용, 환경변수 교체, 기존 자산의 텍스트화) 되돌리기 쉬운 순서로 진행할 수 있다. **시각화·데이터의 견고한 토대 위에 지능 계층을 정렬하는 단계**가 이 프로젝트의 다음 국면이다.
