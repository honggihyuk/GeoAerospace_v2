## 1. 프로젝트 개요

### 1.1 목표
사용자가 **자연어로 명령**하면 LLM이 웹 지도를 제어(특정 위치로 이동, 레이어 토글, 위성/항공기 필터링, 궤도 그리기, 데이터 질의)하고, 항공우주 도메인 데이터(위성 카탈로그·TLE, 항공기 ADS-B, 위성 영상)를 **RAG로 검색**하고 필요 시 **VLM으로 영상을 해석**해 답하는 대화형 지오-플랫폼.

### 1.2 핵심 사용자 시나리오
- "서울 상공을 지나는 저궤도(LEO) 위성만 궤도로 표시해줘" → LLM이 지도 이동 + 위성 필터 + 궤도 링 렌더링
- "지금 태평양 위를 나는 항공기 중 고도 1만 미터 이상만 보여줘" → ADS-B 질의 + 지도 필터
- "ISS의 다음 서울 상공 통과는 언제이고, 최신 위성영상에는 무엇이 보여?" → 궤도 전파(SGP4) + VLM 영상 해석 체인

### 1.3 설계 철학 (가장 중요한 원칙)
> **"VLM이 궤도 데이터를 해석한다"는 표현은 기술적으로 범주 오류(category error)입니다.**
> 궤도 데이터는 이미지가 아니라 **수치 데이터**(TLE, 상태벡터, 케플러 요소)이므로,
> 궤도 계산은 **결정론적 물리 엔진(SGP4/satellite.js)** 이 담당하고,
> **VLM은 오직 위성 "영상"(픽셀)** 해석에만 사용합니다.
> LLM은 이 둘을 조율(orchestrate)하는 **에이전트**로만 동작하며, 궤도 산수를 직접 하지 않습니다.

이 원칙이 전체 아키텍처를 세 개의 특화된 컴포넌트(지식 RAG · 결정론적 천체역학 · 영상 VLM) + 하나의 LLM 조율자로 분리합니다.

## 4. 핵심 기술 영역별 상세 설계

### 4.1 [질문 1 · 최우선] 위성 궤도를 "구형(동그라미) 형태"로 시각화하는 방법

> **용어 정의(요구사항 해석):** "구형(球)"은 **3D 지구본(MapLibre v5 globe)** 을, "동그라미"는 그 위에 그리는 **닫힌 궤도 링(실제로는 타원)** 을 뜻한다. 즉 목표 산출물은 **① 3D 구 지구본 위의 닫힌 궤도 링**(§(4)-A)과 **② 그 지상 투영인 지상궤적**(§(4)-B) 두 가지다. 이것이 본 프로젝트의 **1차 핵심 산출물**이며, 나머지 컴포넌트는 이를 뒷받침한다.

**핵심: TLE → SGP4 전파 → 한 주기 샘플링 → ECI→ECEF→측지좌표 변환 → 폴리라인(닫힌 링)**

#### (1) 데이터: TLE(Two-Line Element)
- TLE는 위성의 평균 궤도요소(경사각, 승교점 적경, 이심률, 근지점 편각, 평균 근점이각, 평균 운동)를 69자 2줄로 인코딩한 것. **일반 케플러 요소가 아니라 SGP4 전용으로 피팅된 값**이다.
- **획득처:** CelesTrak(무료·무인증, 대량 카탈로그, https://celestrak.org/NORAD/elements/) 을 1차, **Space-Track.org**(미 우주군, 권위 있는 원천, 계정 필요)를 원천으로, N2YO(위성별 REST)를 보조로.
- **정확도:** TLE는 하루 약 1~3km씩 오차가 누적 → **매일 재수집** 필요.

#### (2) 전파 엔진: SGP4/SDP4
- 근지구(주기 < 225분)는 SGP4, 심우주(GEO 등)는 SDP4. J2 지구 편평률·대기 항력 등 섭동을 해석적으로 모델링.
- **satellite.js는 서버·클라이언트 양쪽에서 실행**(동일 라이브러리, 역할 분리):
  - **서버(백엔드):** deepagents `propagate_orbit` 도구(§4.5)가 궤도 링/통과 시각 등 **정적 산출물**을 계산해 tool_result로 반환 → 아키텍처 다이어그램(§5)의 "결정론적 천체역학" 블록.
  - **클라이언트:** 위성 현재 위치의 **매 프레임 실시간 전파**(60fps 애니메이션)를 브라우저에서 직접 수행.
  - API: `twoline2satrec(line1, line2)` → satrec, `propagate(satrec, date)` → **ECI** 위치/속도. (정확도 교차검증은 서버 Skyfield/sgp4로.)

#### (3) 궤도 링 그리기 알고리즘 (한 순간이 아니라 한 주기를 샘플링)
```
1. 주기 계산: T(분) = 1440 / n   (n = TLE의 일일 공전수 mean motion)
2. epoch부터 [0, T]를 N개(약 90~360점)로 시간 스텝
3. 각 스텝마다:
   posEci = satellite.propagate(satrec, date).position
   gmst   = satellite.gstime(date)
   posEcf = satellite.eciToEcf(posEci, gmst)          // ECI → ECEF
   geo    = satellite.ecfToGeodetic(posEcf)           // → {lat, lon, height}
   → [lon°, lat°, height] 정점 배열에 push
4. 정점 배열을 폴리라인으로 렌더링
   - deck.gl PathLayer 또는 MapLibre LineString
   - great-circle 보간 활성화, 자오선(±180°) 분할 처리
```

#### (4) 두 가지 산출물 (같은 샘플 루프에서 동시 생성)
- **3D 궤도 링(구형/동그라미):** 고도(height)를 유지 → 우주 공간에 뜬 타원 링. deck.gl에서 고도를 z로, MapLibre globe 위에 오버레이.
- **지상 궤적(ground track):** 같은 샘플을 height=0으로 투영 → 지구 자전 때문에 매 공전마다 서쪽으로 밀리는 특유의 사인파형 경로.

#### (5) 렌더링 배치
- MapLibre globe(구 투영) 베이스 + deck.gl `PathLayer`(궤도 링) + `ScatterplotLayer`(현재 위성 위치).
- 수백~수천 위성은 deck.gl GPU 인스턴싱으로 처리.
- deck.gl은 `@deck.gl/mapbox`의 **`MapboxOverlay`(interleaved 모드, `beforeId` 지정)** 로 MapLibre WebGL2 컨텍스트에 통합 → 궤도 아크가 지구 뒤로 정확히 가려짐(z-buffer 공유, maplibre-gl ≥ 3 필요).
- **3D 지형 병용(§4.9-B):** `setTerrain`(AWS Terrarium DEM)으로 지상궤적·산불·항공 트랙 아래 실제 지형 기복을 표현 → 사실감 향상. globe 투영과 동시 동작.

#### (6) MapLibre v5 Globe 제약과 "실축척 3D 궤도 타원" 대안
MapLibre GL JS **v5.0.0(2025.01)** 이 진짜 3D 구(球) 투영을 제공하지만, 궤도 렌더에는 다음 제약을 설계 단계에서 반영해야 한다.
- **줌 ~12 부근 Mercator 자동 폴백**(셰이더 32비트 float 정밀도 한계): 전체 궤도 조망엔 무해, 단일 위성/항공기 클로즈업 시 유의.
- **fill 레이어 subdivision 상한 128**: 긴 대권 궤도 아크는 **촘촘히 다중 세그먼트로 샘플링**해야 구면에서 매끄럽다((3) 알고리즘의 N을 90→360으로).
- **line/point 피처에 고도 개념 없음**: MapLibre globe만으로는 궤도를 지표 투영으로만 그림. **실축척 3D 궤도 타원**(지표 위 수백~수천 km에 뜬 링)이 핵심 기능이면:
  - **deck.gl `GlobeView`**: `position`에 z(고도) 반영 → 실제 고도로 렌더(단 pitch/bearing 제한, 실험적).
  - **전용 3D 궤도 뷰(R3F 또는 CesiumJS+CZML)를 보조 트랙으로**: 관성계(ECI) 궤도 타원·시간 스크러빙이 필요할 때. 상세 비교·판단 기준은 **§4.6(B)** 참조.

> **참고 문헌(궤도):** Hoots & Roehrich, *Spacetrack Report No.3* (1980); **Vallado et al., "Revisiting Spacetrack Report #3," AIAA 2006-6753 (2006)** — 검증된 SGP4 표준 구현; Chowdhury et al., *OrCo: Web-based SSA Platform*, arXiv:2506.16892 (2025); 시간-동적 시각화는 Zhu, Wang & Li, *Time-Dynamic Objects on Virtual Globes Using CZML*, ISPRS IJGI 7(3):97,102 (2018).

### 4.2 [질문 2] 항공기 궤도(트랙) 실시간 트래킹

**핵심: ADS-B 상태벡터 수집 → 서버 정규화 → 클라이언트 dead-reckoning 보간 → deck.gl IconLayer/TripsLayer**

#### (1) 데이터: ADS-B
- 항공기가 GNSS로 얻은 위치를 초당 1회 1090MHz로 **자발적 방송**. ICAO 24비트 주소, 위경도, 고도, 지상속도, 침로(heading), 상승/하강률 포함. 2020년 이후 대부분 관제공역에서 의무화 → 상업 항공 커버리지 거의 완전.

#### (2) API 선택
| API | 특징 | 용도 |
|---|---|---|
| **OpenSky Network** | 연구용 무료, OAuth2(2025부터), 크레딧 기반 제한 | **1차** 글로벌 상태벡터 |
| **airplanes.live / adsb.lol** | 무료 커뮤니티, 무인증 | **폴백 다중화**(OSIRIS 검증 조합) |
| **adsb.fi** | 무료 커뮤니티, 무인증 JSON, 비상업 | 무료 보조(지역 커버리지) |
| ADS-B Exchange | 무필터(군용 포함), RapidAPI 유료 | 특수 용도 |
| FlightAware AeroAPI | 스케줄·항로 등 풍부, 유료 | 항로 정보 필요 시 |

> **회복탄력 구현(OSIRIS `flights` 차용, §4.8):** 30지역 **팬아웃** + **single-flight** + 90초 캐시 + **429시 15분 쿨다운** + `Promise.allSettled` 폴백, 콜사인/ICAO 타입코드 **정규식 분류**(상용/개인/제트/군용). ⚠️ 위조헤더 한도우회 `stealthFetch`는 **거부**(§4.8-F).

> ⚠️ OpenSky 2025 크레딧 제한 수치(익명 ~400/일, 인증 4,000~8,000/일)는 정책 변경이 잦으므로 **착수 전 라이브 문서로 재확인** 필요.

#### (3) 부드러운 애니메이션: Dead Reckoning(추측 항법)
- 업데이트는 1~10초 간격이지만 렌더는 60fps → 보간 필수.
- 업데이트 사이에는 마지막 heading+지상속도로 **대권 경로**를 따라 전진:
  `newPos = turf.destination(lastPos, speed·Δt, heading)`
- 새 업데이트 도착 시 예측 위치→실제 위치로 lerp/slerp 보간해 점프 제거.
- **Turf.js** 사용: `destination()`, `bearing()`, `along()`, `greatCircle()`, `distance()`.
- `requestAnimationFrame` 루프는 프레임 수가 아닌 **실제 경과 시간** 기반으로.

#### (4) 수천 대 항공기 렌더링 (WebGL)
- **deck.gl `IconLayer`:** GPU 인스턴싱 아이콘, `getAngle`로 heading 회전, 수만 대 확장 가능.
- **deck.gl `TripsLayer`:** `currentTime` uniform으로 잔상(트레일) 페이드 → 최근 비행 궤적 애니메이션에 최적.
- **전략:** 매 프레임 레이어 재생성 대신 **GPU 속성 버퍼(위치/각도)만 갱신**, 뷰포트 밖 컬링, dead-reckoning 계산은 Web Worker로 오프로드.

> **참고 문헌(항공기):** Schäfer et al., "Bringing Up OpenSky," IEEE/ACM IPSN 2014; Hurter et al., "FromDaDy," IEEE TVCG 2009; Buchmüller et al., EuroVis 2015; Sun, *The 1090MHz Riddle* (TU Delft, ADS-B 디코딩).

### 4.3 RAG 파이프라인 (지식 계층)

**원칙: 문서·지식 = RAG 임베딩 / 실시간 궤도·상태 = Tool(API) 호출** (라이브 TLE는 몇 시간이면 낡으므로 임베딩하지 않는다.)

- **말뭉치:** 궤도역학 교재, NASA/NORAD 문서, 미션 설명, 위성 카탈로그 메타데이터.
- **청킹:** 일반 문서는 256~1024 토큰(오버랩). **TLE·카탈로그 레코드는 위성 단위로 원자적 청크**(두 줄을 쪼개면 무의미).
- **검색:** 숫자·ID가 많은 도메인이므로 **하이브리드 검색(dense 임베딩 + BM25 키워드)** 필수. 예: "NORAD ID 25544"는 정확 매칭이 필요.
- **저장:** **pgvector**(카탈로그+임베딩 동일 저장소, HNSW, SQL 메타데이터 필터). 규모/필터 부담 커지면 Qdrant.
- **임베딩 모델(데이터 주권 위해 자체호스팅 오픈 임베더 우선, §4.5-0.5와 정합):** **BGE-M3**(다국어+하이브리드) 권장, E5-large-v2·Nomic-embed 등 HF 오픈모델. 클라우드가 허용되면 `text-embedding-3-large`도 옵션. 강한 범용 임베더 + 하이브리드 키워드가 단일 시맨틱보다 우수.

- **에이전틱 RAG(도구 증강):** "전부 임베딩"이 아니라, 구조화 데이터는 도구호출(§4.5), 문서/영상만 벡터 검색. deepagents 서브에이전트 `gis-query`가 자연어→PostGIS/STAC 공간질의로 번역(text-to-geo-query).

> **참고:** Lewis et al., "Retrieval-Augmented Generation," NeurIPS 2020 (arXiv:2005.11401); **GeoGPT**(arXiv:2307.07930, LangChain 기반 자율 GIS 에이전트 — 본 아키텍처 직접 선례); RS-Agent(arXiv:2406.07089, DualRAG); Agentic RAG survey(arXiv:2501.09136).

### 4.5 LLM 대화형 지도 제어 (조율 계층 — LangChain `deepagents`)

**패턴: ReAct 스타일 에이전트 — 추론 ↔ 도구호출 ↔ 관찰 반복. 구현 하네스는 LangChain `deepagents`.**

#### (0) 왜 deepagents인가
`deepagents`는 "deep agent" 아키텍처(**계획 도구 + 서브에이전트 컨텍스트 격리 + 가상 파일시스템 + 상세 시스템 프롬프트**)를 패키징한 LangChain 라이브러리로, **LangGraph 위의 얇은 하네스**다. 항공우주 분석은 다단계·장기 작업(TLE 수집→전파→영상 해석→지도 렌더)이라 이 4요소가 직접 유효하다.
- **계획(`write_todos`):** 다단계 질의를 상태에 구조화된 TODO로 관리(pending/in_progress/completed).
- **가상 파일시스템(`ls/read_file/write_file/edit_file`):** 대용량 TLE 카탈로그·VLM 캡션 덤프를 컨텍스트 밖으로 오프로드하고 파일명으로 참조.
- **서브에이전트(`task`):** 컨텍스트 격리·병렬. 아래 3종을 정의.
- **LangGraph 이점:** 체크포인팅·토큰 스트리밍·human-in-the-loop 인터럽트·세션별 스레드 영속 → LangGraph Server로 배포하면 웹 플랫폼에 그대로 적합.

#### (0.5) 에이전트 백본 모델 — Hugging Face 오픈모델 Qwen3 (자체호스팅)
백본은 **Claude Tool use가 아니라 자체호스팅 HF 오픈모델**을 사용한다(데이터 주권·비용·오프라인 운용). deepagents는 모델이 **구조화된 tool_calls를 방출**하기만 하면 동작하므로, OpenAI 호환 함수호출을 지원하는 오픈모델이면 된다.

**선정: Qwen3 계열 (Apache-2.0).** Qwen3(2025.04)은 Hermes 스타일 도구 템플릿을 내장하고 하이브리드 thinking/non-thinking을 지원하며, 오픈모델 중 에이전트/도구호출에서 선두권. 규모별 티어:
| 티어 | 모델 | HF id | 용도 / GPU(대략) |
|---|---|---|---|
| **로컬 개발(현 PC)** | **Qwen3-8B AWQ/GPTQ 4-bit** | `Qwen/Qwen3-8B` (+ 4-bit quant) | 에이전트 그래프 구축·디버그. **RTX 3060 12GB에 4-bit ~6–8GB로 적재**(아래 프로파일) |
| 프로덕션(권장) | **Qwen3-30B-A3B**(MoE) | `Qwen/Qwen3-30B-A3B` | 3B만 활성→8B 속도로 대형급 계획력. bf16 ~60GB → **클라우드/상위 GPU 필요**(현 로컬 PC 불가) |
| 프로덕션(dense) | **Qwen3-32B** | `Qwen/Qwen3-32B` | dense 예측성 선호 시. FP8 ~34GB, AWQ ~18–20GB → **상위 GPU** |

> ⚠️ **8B는 개발엔 충분하나 다단계 오케스트레이션 신뢰성은 marginal**(도구 포맷 이탈 발생). 프로덕션은 **Qwen3-30B-A3B** 권장. 대안 벤치마크 후보: **Mistral-Small-3.2-24B**(Apache-2.0, FC 템플릿 개선 + 멀티모달로 VLM 겸용 가능), **GLM-4.5-Air**(BFCL 상위 오픈모델), **Llama-3.3-70B**. ⚠️ xLAM-2는 cc-by-nc(비상업)로 제외, watt-tool은 Llama 베이스 라이선스 확인 필요.

#### (0.6) 로컬 개발 PC 프로파일 — RTX 3060 12GB + 32GB RAM (확인됨, GPU 예산 추후 조정)
현 개발 PC는 **NVIDIA RTX 3060(전용 VRAM 12GB, Ampere) + 32GB RAM**. 이 사양의 제약과 처방:
- **bf16 8B(~16GB)는 12GB에 미적재** → **4-bit 양자화 필수**. **Ampere는 FP8 미지원**(FP8=Ada/Hopper) → **AWQ 또는 GPTQ INT4** 사용(FP8 아님).
- **Qwen3-30B-A3B는 4-bit라도 ~18GB로 12GB 초과 → 로컬 구동 불가**. 프로덕션 티어는 클라우드/상위 GPU에 배포하고, 로컬은 8B 4-bit로 에이전트 로직만 검증.
- **VLM(GeoChat 7B) 동시 상주 불가**: 12GB에 8B 백본 + 7B VLM 동시 적재는 어려움 → **로컬 개발 시 VLM은 HF Inference Endpoints(클라우드)로 분리**하거나 필요 시에만 교대 로딩.

**로컬 서빙 A — vLLM(4-bit, 프로덕션과 동일 스택):**
```bash
vllm serve Qwen/Qwen3-8B-AWQ \        # ⚠️ 공식/커뮤니티 AWQ repo id는 HF에서 확인
  --quantization awq \
  --max-model-len 16384 \             # 12GB에 맞춰 컨텍스트 상한(KV 캐시 여유 확보)
  --gpu-memory-utilization 0.90 \
  --enable-auto-tool-choice --tool-call-parser hermes --reasoning-parser deepseek_r1
```
**로컬 서빙 B — Ollama(가장 간편, 개발 편의):** `ollama run qwen3:8b`(Q4_K_M ~5GB). OpenAI 호환 `/v1`로 tool_calls 지원 → deepagents에 `ChatOpenAI(base_url="http://localhost:11434/v1")`로 동일하게 주입 가능. 초기 개발·디버깅에 권장, 프로덕션은 vLLM.

- **VRAM 예산(대략, 12GB 기준):** 4-bit 가중치 ~6GB + KV 캐시 ~4GB(≈16K 컨텍스트) → 안정. 컨텍스트를 32K로 늘리면 `--max-model-len`/동시성 하향 조정.
- **RAM 32GB**: PostGIS/pgvector·인제스트 파이프라인·프론트 빌드 동시 구동에 충분.
> **GPU 예산은 조정 가능 항목.** 프로덕션 전환 시 Qwen3-30B-A3B(≥24–48GB GPU 또는 관리형 엔드포인트)로 승급하고, 로컬은 8B 4-bit로 유지하는 **하이브리드**(로컬 개발 = 8B, 배포 = 30B-A3B)를 기본 운영 모델로 한다.

**서빙 공통 플래그(vLLM, OpenAI 호환) — 프로덕션 기본(전체 정밀도):**
```bash
vllm serve Qwen/Qwen3-30B-A3B \      # 프로덕션. 로컬 개발은 §4.5-0.6의 4-bit 변형(서빙 A)
  --enable-auto-tool-choice \
  --tool-call-parser hermes \        # Qwen3 dense/MoE 공통 (Coder 계열만 qwen3_coder)
  --reasoning-parser deepseek_r1     # thinking 모드 유지 시 <think> 분리
```
> 위 3개 파서 플래그는 로컬(§4.5-0.6 서빙 A)·프로덕션 공통. ⚠️ reasoning+tool_choice="required" 동시 사용 시 vLLM 파싱 버그 사례(#19051,#19513) → 도구 선택 턴에는 `/no_think`로 thinking 끄는 것을 검토. Qwen-Agent로 라우팅할 경우엔 위 파서 플래그를 빼야 하나, LangChain/deepagents는 OpenAI형 tool_calls를 기대하므로 **플래그 유지**가 맞음.

agent = create_deep_agent(

        {"name": "orbit-analyst",  "description": "TLE 수집·SGP4 전파·통과 계산",

#### (1) 도구 설계
- **도구를 JSON 스키마로 노출** (God-function 대신 좁은 다수 도구. 이름은 위 코드와 일치 — **snake_case 통일**):
  - **지도 제어:** `fly_to({lng,lat,zoom,pitch})` · `set_filter({layer_id,expression})` · `add_layer({id,type,source,paint})` · `draw_orbit({norad_id})`(§4.1 궤도 링 렌더 트리거) · `query_data({layer,bbox,predicate})`
  - **데이터·물리(결정론적):** `get_tle({norad_id})` · `propagate_orbit({tle,time})` · `query_aircraft({bbox,filters})`
  - **RAG·VLM:** `retrieve_docs({query})` · `analyze_image({image,question})`
  - **산불(§4.7):** `filter_fires({region,date,day_range,min_frp,...})`
- **아키텍처: 서버측 에이전트 + 클라이언트측 실행**
  - **서버:** `deepagents` 루프를 **LangGraph Server**로 배포, 시스템 프롬프트+도구 스키마 보유, LangGraph 스트리밍으로 토큰·이벤트 전송. DOM 접근 안 함.
  - **클라이언트:** `maplibregl.Map` 인스턴스와 tool executor 보유. `tool_use` 블록을 실제 지도 호출로 매핑하고 `tool_result`(성공+질의결과) 반환.
  - **상태 동기화:** center·zoom·활성 레이어·필터·가시 피처 ID를 담은 **canonical map-state 객체**를 매 턴 컨텍스트에 주입 → 모델이 가정이 아닌 **실제 상태**로 추론.
- **MCP 활용:** 데이터 검색 도구(TLE·ADS-B·PostGIS)는 **MCP 서버**로 래핑해 재사용성 확보. 지도 조작 도구는 라이브 `Map` 객체가 필요하므로 클라이언트에 유지.
- **환각 방지(그라운딩):**
  1. 알려진 개체의 좌표를 LLM이 직접 생성하지 못하게 하고 `geocode()`/`lookupEntity()` 도구 강제.
  2. 뷰포트·가시 피처 수·질의 결과를 tool_result로 주입.
  3. "일본 상공 위성 몇 개?" 류는 반드시 `query_data()`로 실제 데이터셋 조회.
  4. 도구 인자 스키마+경계 검증 후 실행, 무효 시 재프롬프트.

> **참고:** Yao et al., "ReAct," 2022 (arXiv:2210.03629); Schick et al., "Toolformer," 2023 (arXiv:2302.04761); Roberts et al., "GPT4GEO," 2023 (arXiv:2306.00020, LLM의 좌표 환각 증거); Manvi et al., "GeoLLM," ICLR 2024 (arXiv:2310.06213); Zhang et al., "MapGPT," Cartography and GIS 2024 (가장 근접한 선행 연구).

### 4.6 Three.js 3D 렌더링 방안 (glTF 위성·궤도 튜브·센서 콘)

deck.gl(§3)이 대량 포인트/라인 GPU 렌더를 담당한다면, **Three.js**는 "정교한 단일 3D 객체" — 실제 위성 glTF 모델, 부피 있는 궤도 튜브, 센서 풋프린트 콘 — 를 담당한다. 두 경로를 병행한다.

#### (A) MapLibre v5 globe에 Three.js를 합성 — 기본 (공식 지원 확인됨)
MapLibre `type:'custom'` 레이어로 `THREE.Scene`을 지도 카메라와 동기화해 **베이스맵 위에 직접** 3D 위성을 얹는다. **핵심: v5 globe 투영에서 공식 지원된다**(첫 파티 예제 존재).
- API: `map.addLayer({ id, type:'custom', renderingMode:'3d', onAdd(map,gl){...}, render(gl,args){...} })`
  - `onAdd`에서 `THREE.WebGLRenderer({ canvas: map.getCanvas(), context: gl })`로 지도의 WebGL2 컨텍스트 공유.
  - **위치(globe-safe):** `map.transform.getMatrixForModel(lngLat, altitude)` — "현재 투영과 무관하게 동작"(globe/Mercator 공통). Mercator 전용 `MercatorCoordinate` 수학을 대체.
  - **투영행렬:** `args.defaultProjectionData.mainMatrix` → `new THREE.Matrix4().fromArray(...)`를 모델행렬과 곱해 `camera.projectionMatrix`에 설정.
  - `renderingMode:'3d'`로 globe depth 버퍼 확보. 애니메이션은 `map.triggerRepaint()`.
- 예제: [add-a-3d-model-to-globe-using-threejs](https://maplibre.org/maplibre-gl-js/docs/examples/add-a-3d-model-to-globe-using-threejs/) (globe), [add-a-3d-model-using-threejs](https://maplibre.org/maplibre-gl-js/docs/examples/add-a-3d-model-using-threejs/) (Mercator).
- ⚠️ **주의:** globe 커스텀 레이어는 Mercator 경로보다 최신·생태계가 얇음. **threebox 등 서드파티 래퍼는 globe 미완성**(maintainer HarelM: "globe는 추가 작업 필요"; threebox raycasting 깨짐) → **raw Three.js**를 쓰고, 실제 궤도 고도에서 depth/scale 튜닝 시간을 확보. v4→v5 브레이킹: 단일 `matrix` 인자 → `args.defaultProjectionData` API로 마이그레이션 필수.

#### (B) 별도 "우주 뷰"(실축척 3D) — R3F / CesiumJS *(문서 내 CesiumJS 논의는 여기로 통합, 로드맵 P2.7 착수)*
베이스맵과 분리된 스탠드얼론 3D 씬(지구 + 실제 고도 궤도 리본 + 센서 콘)을 **정식 착수 범위**(§6 P2.7)로 구현한다. 두 선택지다. MapLibre globe는 line/point에 고도 개념이 없어(§4.1-6) 실축척 3D 궤도 타원 표현이 어렵기 때문에, 이 요구가 커지면 **MapLibre(2D 분석면·베이스맵) + 전용 3D 궤도 뷰**의 **2트랙** 구성을 택한다.

- **react-three-fiber(R3F) + `r3f-globe`**(react-globe.gl 저자, three-globe 엔진): 경량·완전 커스터마이즈·React 통합. 브랜드화된 스타일 씬에 유리. (A)의 Three.js 자산을 그대로 재사용.
- **CesiumJS + CZML(물리 정확 궤도에 유리, 위험 최소 옵션):**
  - **관성계(ECI) 네이티브 지원**(`ReferenceFrame.INERTIAL`): 궤도 타원을 관성계에 직접 그리고 지구를 그 아래에서 자전시킴 — **궤도를 구 위 타원으로 보이는 가장 자연스러운 방식**. MapLibre/deck.gl에는 관성계 개념이 없어 매 프레임 ECEF/측지 변환이 필요한 것과 대비.
  - 정확한 **WGS84 측지** + **시간동적 CZML**(내장 타임라인/클럭, `SampledPositionProperty` 보간)을 기본 제공. `tle2czml`로 TLE→CZML 직변환 가능.
- **판단 기준:** 스타일·React 통합·경량 우선 → R3F. 물리 정확도·시간 스크러빙·관성계 궤도 우선 → CesiumJS. 어느 쪽이든 **MapLibre가 1차 베이스**이고 이 뷰는 보조 트랙이다.

> CesiumJS의 저수준 프리미티브 렌더링 API는 우리 스택(MapLibre+deck.gl)에서 채택하지 않는다(등가는 deck.gl GPU 인스턴싱, §4.8-F). CZML 시간-동적 시각화 근거 논문은 Zhu et al. 2018(§4.1 참고문헌·§8).

#### 렌더 기법 & 오픈소스 레퍼런스
- **궤도:** SGP4 샘플 점 → `THREE.TubeGeometry`(부피 튜브) 또는 `Line2`/`LineGeometry`(월드단위 두께 리본; 일반 `THREE.Line`은 linewidth 무시).
- **센서 콘:** `THREE.ConeGeometry`를 위성에 앵커, nadir/타깃 지향, 반투명. (WWV **FrustumRenderer**가 동일 개념의 검증 구현 — §4.8-C)
- **LOD 승격(WWV ModelManager 차용, §4.8-C):** 원거리/저줌=아이콘·빌보드, 근거리/고줌=**glTF 3D 모델**로 승격 → 대량 위성에서 성능·가독성 양립. **StackClustering**으로 Starlink 등 밀집 군집을 저줌에서 분산 표시.
- **대안(선택):** 대량 인스턴싱·내장 피킹이 필요하면 deck.gl `ScenegraphLayer`/`SimpleMeshLayer`(glTF 2.0). 단 ⚠️ deck.gl 메시의 MapLibre **globe** 정합은 별도 검증 필요(deck.gl `_GlobeView`는 MapLibre globe와 자동 정렬 아님).
- 레퍼런스: [dsuarezv/satellite-tracker](https://github.com/dsuarezv/satellite-tracker)(Three.js+React+satellite.js), [satvisorcom/satvisor](https://github.com/satvisorcom/satvisor), [react-globe.gl satellites 예제](https://github.com/vasturiano/react-globe.gl).

> **채택:** 위성/궤도를 **MapLibre 베이스맵 위에 합성** → (A) Three.js custom layer(P2.5). **전용 실축척 우주 뷰**(B) R3F/`r3f-globe` 또는 CesiumJS는 **로드맵 P2.7로 착수 범위에 포함**(§6). 대량 마커·아크는 계속 deck.gl(§3).

### 4.7 NASA GIBS + FIRMS 산불 모니터링 통합 (권장 — 채택)

`worldview-main/doc/2026-07-14_gibs_firms_기술구현분석.md` 분석 결과, **우리 프로젝트에 매우 적합하여 정식 기능으로 채택**한다. 근거: GIBS/FIRMS의 역할 분리가 우리의 **"시각화=타일 / 구조화 데이터=도구호출"** 원칙(§4.3)과 정확히 일치하고, 산불 모니터링은 위성·영상·지상데이터를 한 시나리오에 결합하는 **킬러 유스케이스**다.

#### 핵심 통찰: 같은 산출물의 두 표현
| 축 | **GIBS**(래스터/벡터 타일) | **FIRMS**(벡터 포인트) |
|---|---|---|
| 데이터 모델 | 격자 픽셀(값이 구워짐) | 1탐지=1피처(FRP·confidence 속성) |
| 서빙 | 타일 피라미드 WMTS(CDN 캐시, 초저지연) | 질의 API(REST, bbox·날짜·속성 필터) |
| 적합 작업 | **"보여줘"**(시각화·비교·애니메이션) | **"필터/알려줘"**(위치·개수·강도) |
| 에이전트 도구 | `add_layer`(GIBS 래스터) | **`filter_fires`(신규, FIRMS 포인트)** |

> 즉 **GIBS=시각화, FIRMS=분석/필터**. 두 도구로 역할 분리해 에이전트에 노출.

#### 신규 도구: `filter_fires` (서버측 deepagents 도구)
```python
@tool
def filter_fires(region, date, day_range=1,
                 source="VIIRS_SNPP_NRT",
                 min_confidence=None, min_frp=None, fire_type=0):
    """FIRMS Area API로 산불 탐지 포인트를 bbox·기간·속성 필터해 반환."""
    bbox = geocode_region(region)                 # 지오코딩(Nominatim/OSM 우선, 오픈·토큰프리) + SSRF 가드(§4.8-B)
    # GET /api/area/csv/{MAP_KEY}/{SOURCE}/{W,S,E,N}/{DAY_RANGE}/{DATE}
    #   MAP_KEY는 백엔드 .env에만(프론트 노출 금지), 5000요청/10분 한도
    rows = fetch_firms_area_csv(bbox, date, day_range, source)
    fires = filter(rows, min_confidence, min_frp, fire_type)  # type 0=초목화재
    return {"points": [{lat,lon,frp,confidence,acq_datetime}], "summary": ...}
    # 프론트: deck.gl ScatterplotLayer(MapLibre 오버레이) 마커, 클릭 시 FRP·confidence 팝업
```
- **인증/한도:** `MAP_KEY`는 **백엔드에만** 은닉(프론트 노출 금지), 5000요청/10분. `search_layers`와 동일한 서버측 도구 패턴(외부 키 은닉, tool_result 회신).
- **소스:** `VIIRS_SNPP/NOAA20/NOAA21_NRT`, `MODIS_NRT`, `LANDSAT_NRT` 등. 속성 필터 근거: `confidence`(확실성), `frp`(MW, 강도), `type`(0=초목/1=화산/2=정적/3=해상).
- **구현 디테일(OSIRIS `fires` 차용, §4.8):** **VIIRS→MODIS 폴백**(고해상 우선, 실패 시 대체), **NASA EONET 화산** 병합, 의존성 없는 **자체 CSV 파서**, 대영역 응답은 **~2000점 샘플링**으로 전송량·렌더 부담 제어. `Promise.allSettled`로 소스 장애 격리.

#### 3계층 시너지 (우리 플랫폼 고유 가치)
산불 질의 하나가 세 컴포넌트를 엮는다:
1. **FIRMS 포인트**(`filter_fires`) → "어디서 얼마나 강하게"(정확 위치·FRP).
2. **GIBS 래스터**(`add_layer`, `TIME=`) → 맥락 영상(MODIS/VIIRS TrueColor, 연기 플룸)을 아래 오버레이.
3. **VLM**(`analyze_image`, §4.4) → GIBS 영상 타일을 GeoChat으로 해석("연기 확산 방향/규모").
4. (선택) **궤도**(§4.1) → 어느 위성이 이 탐지를 언제 촬영했는지 통과 시각.
→ `fire-analyst` 서브에이전트(§4.5)가 이 체인을 오케스트레이션.

#### 예시 시나리오
> "지난 3일 캘리포니아 산불 중 FRP 100MW 이상만 지도에 표시하고, 최신 위성영상으로 연기 확산을 설명해줘"
> → `filter_fires(region="California", day_range=3, min_frp=100, fire_type=0)` (FIRMS 포인트 마커)
> → `add_layer("MODIS_Terra_CorrectedReflectance_TrueColor", TIME=today)` (GIBS 맥락영상)
> → `analyze_image(gibs_tile, "연기 플룸의 방향과 규모는?")` (VLM 해석)

> **구현 정합성:** GIBS RESTful 타일 URL은 `…/wmts/epsg3857/best/{Layer}/default/{Time}/{TileMatrixSet}/{z}/{y}/{x}.{fmt}` 형식이며 MapLibre `raster` 소스로 직접 소비. FIRMS는 CSV/GeoJSON을 벡터 레이어로. (분석 문서 §1.2, §2.3 참조)

### 4.8 레퍼런스 코드베이스 채택 기술 분석 (WorldWideView · OSIRIS 구현 분석)

두 프로젝트의 **실제 구현 분석 문서**(`큐브위성사업/기술분석/0716_WWV_*`, `0716_OSIRIS_*`)를 검사해, 우리 도메인과 겹치면서 **현 제안서 대비 우수하거나 보완적인** 구현 기술을 선별했다. 활용도·우수성 기준으로 **채택**하고, **열등하거나 ToS 위반 소지가 있는 기술은 명시적으로 거부**한다.

#### (A) 채택 — 실시간 데이터 파이프라인 & 프론트 성능
| 기술 | 출처 | 활용도/우수성 | 우리 적용 |
|---|---|---|---|
| **`useRef` + 수동 리렌더**(`setDataVersion(v=>v+1)`) | OSIRIS | 수천 엔티티를 React state에 안 넣어 리렌더 비용 회피. 우리 위성/항공 고빈도 갱신에 **직결** | deck.gl 레이어 데이터를 `dataRef.current`에 병합, 버전 카운터로만 트리거 |
| **레이어 인식 지연 로딩**(`layerFetchedRef` Set, 켤 때 1회) | OSIRIS | 중복 요청 방지, 초기 대비 엣지요청 75% 감축 | 레이어 토글 시 최초 1회 페치 |
| **차등 폴링 + `document.hidden` 가드** | OSIRIS | 데이터 특성별 간격(AIS 10s/항공 5min/지진·뉴스 15~30min), 백그라운드 탭 스킵 | 위성 TLE 일1회, ADS-B 5~10s, FIRMS 온디맨드 |

- **배포**(WWV/OSIRIS): Next.js **standalone** + Docker **멀티스테이지**(non-root) + **docker-compose**(app/db/redis) + **nginx 타일 캐시 프록시**(GIBS/CARTO 타일 1년 캐싱, thundering-herd lock). → 인프라 구체화.

#### (F) 거부 — 열등하거나 부적합/위험 (명시적 제외)
| 기술 | 출처 | 거부 사유 |
|---|---|---|
| **자체 간이 SGP4**(케플러+GMST) | OSIRIS | ❌ 정확도 열등. 우리는 **Vallado 표준 SGP4/SDP4(satellite.js/Skyfield)** 유지(§4.1). 간이 구현은 섭동 미반영으로 오차 큼 |
| **`stealthFetch`**(UA 로테이션 + **위조 `X-Forwarded-For`** 로 업스트림 레이트리밋 회피) | OSIRIS | ❌ **API ToS 위반·우회(abuse) 소지**. 채택 금지. 정당한 방법(캐시·백오프·다중 소스·유료 티어)으로 한도 관리 |
| **DB 없이 `globalThis` 캐시만** | OSIRIS | ⚠️ 부분 부적합. 우리는 **RAG용 pgvector+PostGIS 필수** → DB 유지하되, 핫 실시간 데이터에만 globalThis/Redis TTL 병용(하이브리드) |
| **CesiumJS 프리미티브 컬렉션 API** | WWV | △ 원리(저수준 GPU)만 차용. 우리 렌더는 MapLibre+deck.gl이므로 Cesium 전용 API 대신 **deck.gl GPU 인스턴싱**으로 등가 달성 |

> **요약:** WWV에서는 **실시간 파이프라인·3D 성능·MCP·품질게이트**, OSIRIS에서는 **회복탄력 팬아웃·캐싱·SSRF 보안·데이터소스 다중화·통합 엔티티 모델**을 채택한다. 두 코드베이스의 **간이 SGP4·stealthFetch·DB-less**는 우리 정확도/보안/RAG 요구에 미달하여 거부한다.

### 4.9 Google Earth Enterprise · Mapbox GL JS 기술 채택 분석

두 레퍼런스(`google/earthenterprise`, `mapbox/mapbox-gl-js`)를 **라이선스 정합성** 관점에서 검사했다. 우리 방향(오픈소스·토큰프리·데이터 주권)과 **일관되면서 우수한 기술만 채택**하고, 독점/폐기된 것은 거부하되 **핵심 우수 역량인 "3D 지형(terrain)"** 은 오픈 DEM으로 도입한다.

#### (A) 렌더러 — MapLibre 유지, Mapbox GL JS 거부 (라이선스 결정적)
- **Mapbox GL JS v2.0(2020.12)+ 는 독점**: Mapbox TOS로 재라이선스 → **활성 계정·액세스 토큰 강제, 텔레메트리/과금 코드 수정 금지, 위반 시 자동 종료**. v3도 동일. → **우리 오픈/토큰프리 요구를 정면 위배 → 렌더러 전환 거부.**
- **MapLibre GL JS는 Mapbox GL JS 마지막 오픈 릴리스(v1.13, BSD-3-Clause)의 커뮤니티 포크** → 우리는 이미 올바른 선택을 하고 있음. **MapLibre v5 유지.**
- **단, Mapbox 기원의 오픈 표준/기법은 채택**(MapLibre가 이미 구현): **MVT**(벡터타일, 스펙 CC-BY·코드사용 무제약)·**Style Spec**(MapLibre 포크)·**TileJSON**·**MBTiles**·**Terrain-RGB/Terrarium 인코딩**. v3의 Standard 스타일 수준 비주얼(3D 조명/그림자)은 자체 스타일 + 오픈 DEM 지형으로 ~80% 근접, 락인 0%.

#### (B) 3D 지형(terrain) — 두 프로젝트가 개척한 핵심 우수 기술, **오픈으로 채택**
Google Earth와 Mapbox가 공통 개척한 지형 렌더링을 **MapLibre v5 네이티브**로 도입한다. 궤도 지상궤적·산불·항공 트랙 아래 실제 지형 기복을 표현해 §4.1 시각화의 사실감을 높인다.
- **API:** `map.addSource(dem, {type:'raster-dem', tiles:[...], encoding:'terrarium'|'mapbox'})` → `map.setTerrain({source:dem, exaggeration:1.5})` + `hillshade`/`sky` 레이어, `projection:'globe'`와 동시 동작.
- **오픈 DEM 소스(토큰프리 우선):**
  | 소스 | 엔드포인트/형식 | 개방성 |
  |---|---|---|
  | **AWS Terrain Tiles(Terrarium)** — 기본 채택 | `s3.amazonaws.com/elevation-tiles-prod/terrarium/{z}/{x}/{y}.png` (256px) | ✅ **완전 오픈·무인증** 드롭인 `raster-dem`. ⚠️ Mapzen 유래, SLA 없음(정적 S3) |
  | **Copernicus DEM GLO-30** — 프리미엄(자체 타일링) | S3 `copernicus-dem-30m`(COG, 1°×1°) | ✅ 무료(ESA/Copernicus). ⚠️ 사전타일 아님 → **직접 raster-dem 타일 생성**(=데이터 주권 강화) |
  | USGS/SRTM | OpenTopography/USGS | ✅ 공공(미정부), 자체 타일링 필요 |
  | MapTiler terrain | `api.maptiler.com/...?key=` | ⚠️ **토큰 필요 → 코어 제외**(옵션) |

#### (C) Google Earth Enterprise — 실행 컴포넌트 거부, 아키텍처 참조만
- **상태:** GitHub **2023.01 아카이브(읽기전용), 최종 릴리스 2021, 지원 OS 전부 EOL(CentOS 6/7 등)** → **죽은 프로젝트, 자체호스팅 거부.** (Apache-2.0이긴 함.)
- **현대 대체:** GEE의 이미지/타일 서버 역할은 **NASA GIBS WMTS**(오픈·토큰프리, §4.7)로, 3D 콘텐츠는 **OGC 3D Tiles**로 대체 완료. GEE는 **아키텍처(쿼드트리 타일링·지형 메시·포터블 글로브) 학습용 참조**로만.
- **채택할 개방 포맷:** **KML/KMZ**(OGC 표준) — 정적 오버레이·풋프린트 **가져오기/내보내기**·상호운용에 도입. (⚠️ KMZ는 사실상 표준; ⚠️ GEE `.glb`는 glTF `.glb`와 다른 포맷 — 이름 충돌 주의.)

#### (D) 선택적 프리미엄 (3D 트랙 전용, 명시적 옵션 — 코어 의존 금지)
| 항목 | 용도 | 라이선스/비용 |
|---|---|---|
| **Google Photorealistic 3D Tiles**(OGC 3D Tiles) | 도시 실사 3D 메시. deck.gl `Tile3DLayer`(≥8.9.13)/CesiumJS 소비, MapLibre 직접 불가 → §4.6-B 3D 트랙 | ⚠️ **API 키+과금** (~$6 CPM, 월 1,000 이벤트 무료). 저작권 표시 의무 |
| **Cesium World Terrain / Ion**(quantized-mesh) | §4.6-B CesiumJS 트랙 고정밀 지형 | ⚠️ **Ion 토큰 필요**. 무료=비상업, 상업 ~$149/월. 자체 quantized-mesh 생성으로 토큰프리화 가능 |
| **CZML**(시간동적 궤도) | 움직이는 위성 궤도 표시(Cesium 트랙). `tle2czml`로 TLE→CZML | ✅ 오픈 포맷(Cesium 생태계, MapLibre 비네이티브) → §4.6-B |

> ⚠️ Google P3DT/Cesium Ion 가격은 벤더가 주기적으로 개정 → **예산 확정 전 재확인.**

#### 요약 — 채택 vs 거부

### 4.10 UX/UI 디자인 방향 — "Orbital Command" 시각 컨셉

> **핵심 방향성:** 본 플랫폼의 최대 차별점 중 하나는 **"시각적으로 압도적으로 보여야 한다"**. 기능이 아니라 **첫인상**이 제품의 격을 결정한다. GEE·Mapbox·최상위 SSA 툴의 시각 언어를 근거로 **시네마틱 다크 관제 콘솔(command deck)** 을 지향한다.
> **디자인 산출물(게시됨):** ① 화면 목업(라이브 글로브) — [claude.ai/code/artifact/2dc57f9d](https://claude.ai/code/artifact/2dc57f9d-1633-473a-ae53-73d3f0d15312) · ② **컴포넌트 라이브러리(디자인 시스템)** — [claude.ai/code/artifact/35ca13aa](https://claude.ai/code/artifact/35ca13aa-90de-4cbc-846b-162114923c7e)

#### 디자인 원칙 (5)
1. **Map-first, full-bleed.** 글로브가 유일한 주인공. 모든 컨트롤은 글라스 패널로 **부유**하고, 비필수 크롬은 자동 은닉. 지도를 가리는 split-pane 금지 → AI 챗은 **오버레이 드로어**(지도는 뒤에서 계속 인터랙티브).
2. **Cinematic globe.** "비싸 보이는" 느낌은 **대기 산란(Rayleigh+Mie) 림 글로우 + 골든아워 조명 + 로그-이징 fly-to 카메라 + 별필드**에서 나온다(GEE 방식). 정적 지도가 아니라 **살아 있는 행성**.
3. **Instrument, not app.** 텔레메트리·좌표·시각은 **모노스페이스+tabular-nums**로 "계기판" 질감. 대문자 마이크로 라벨(자간 넓힘)로 관제 HUD 톤.
4. **Restraint = premium.** 90% 다크 중립 / 8% 시안(데이터·궤도) / 2% 앰버·상태색. 액센트를 한 곳에만 집중 → "게이머 RGB"가 아닌 "관제 콘솔". 궤도만 **선택 시 렌더**(전부 그리면 노이즈).
5. **의도적 단일 다크 월드.** 실제 관제실은 항상 어둡다 → 다크 전용을 **선택**으로 확정(라이트 테마 생략은 누락이 아닌 결정).

#### 시각 언어 (디자인 토큰)
- **컬러 "Midnight × Golden Horizon":** 우주흑 `#05070F` · 네이비 패널 `#0E1728` · 그리드 `#1E2A44`(navy-biased 중립) · **주 액센트 아이스 시안 `#5CE1FF`**(궤도·데이터·인터랙션) · **시네마틱 앰버 `#FFB74D`**(수평선 글로우·추적 대상, 극소량) · 상태 녹`#34D399`/황`#FBBF24`/적`#F87171`.
- **타이포:** UI=정제된 시스템 산세리프(대문자 라벨+자간) / 데이터·좌표·텔레메트리=모노스페이스(tabular-nums). 웹폰트 CDN 대신 시스템 스택으로 폴백 리스크 제거.
- **글라스 패널:** `backdrop-blur` + 네이비 틴트 + 시안 1px 림(흰 보더 지양). 텍스트 뒤 미세 scrim으로 지도 위 가독성(AA) 확보.
- **모션:** 글로브 자전·궤도 위성 이동·별 트윙클·티커. `prefers-reduced-motion` 존중(정지 프레임). 과잉 애니메이션은 "AI 생성" 인상 → 절제.

#### 화면 구성 (관제 데크 레이아웃)
- **상단 명령 바:** 브랜드 + **자연어 명령 입력(⌘K)** + 상태 칩(LIVE·OBJECTS·TLE FRESH·UTC 클럭).
- **좌측 레이어 레일:** 위성/궤도/항공기/산불/센서 FOV/3D 지형 토글(글라스 칩·카운트) + 범례.
- **추적 대상 HUD 카드:** 코너 브래킷 프레임 + 모노 텔레메트리(ALT·VEL·INCL·PERIOD·NEXT PASS). 앰버 강조.
- **우측 AI 챗 드로어:** GeoAgent(Qwen3·deepagents) 대화 + **에이전트 계획 스텝 노출**(get_tle→propagate_orbit→draw_orbit) + 결과 칩. §4.5와 직결.
- **하단 타임 스크러버:** −과거 · **NOW** · +예측, 재생/속도(60×) — 궤도 전파·통과 예측용(§4.1).
- **텔레메트리 티커 + 코너 레티클 + 측정 그리드** 오버레이.

#### 시그니처 시각 기법 (WebGL/deck.gl/Three.js)
- **Bloom 후처리**(`UnrealBloomPass` 또는 deck.gl `PostProcessEffect`) — 궤도·위성만 선택 발광(2-composer 마스크). #1 시네마틱 업그레이드.
- **가산 블렌딩 궤도 링**(`AdditiveBlending`) — 교차 지점이 밝아지는 SSA 특유의 발광 메시.
- **페이딩 트레일**(정점 알파 램프) · **프레넬 림 글로우**(대기) · **GPU 별필드** · **거리 기반 포그**(원거리 궤도 후퇴 → 깊이감).
- MapLibre `type:'custom'` 레이어에 이 발광 렌더러를 주입(§4.6-A). keeptrack.space가 브라우저 60fps·5만+ 객체 상한을 입증.

#### 레퍼런스 (에이전시 덱용 벤치마크)
- **시네마틱 글로브:** Google Earth / Earth Studio(대기·골든아워·로그이징), Mapbox Standard(Day/Dusk/Dawn/Night 조명·fog·글로브 atmosphere).
- **SSA 시각:** **keeptrack.space**(오픈소스, 발광 궤도·트레일·색분류), NASA Eyes, satellitemap.space(Starlink train), Slingshot Beacon·LeoLabs·Saber(관제 미학).
- **관제 HUD:** Palantir Gotham류 다크·HUD 프레이밍·글라스모피즘·모노 텔레메트리.

> **디자인 산출물(에이전시 단계):** ① 무드보드·레퍼런스 덱 ② **디자인 토큰·컴포넌트 라이브러리 20종**(파운데이션+컨트롤+데이터/상태+에이전트/시스템 — 위 Artifact에 상세) ③ 하이파이 목업(주요 화면 3~5) ④ 모션 스펙(fly-to 이징·bloom·트레일) ⑤ 접근성(다크 AA 대비·reduced-motion·키보드 포커스). 구현 이관 = React 컴포넌트 + CSS 변수 토큰.

## 7. 리스크 및 스코핑 (현실 vs 과장)

**현실적으로 구현 가능:**
- RAG 기반 궤도역학·카탈로그 Q&A (고가치·저위험)
- LLM이 SGP4/Skyfield를 호출하고 결과를 자연어로 설명하는 **조율자** 역할
- GeoChat급 VLM의 위성 **영상** 해석(장면 분류·객체/변화 탐지)

**과장/비현실 (피할 것):**
- ❌ "VLM이 궤도 데이터를 해석" → 범주 오류. 궤도는 수치 → SGP4 도구가 담당, VLM은 실제 영상에만.
- ❌ LLM이 raw TLE로 궤도를 "머릿속에서" 계산 → 반드시 환각. 항상 SGP4에 위임.
- ❌ 단일 모델이 영상+궤도역학을 end-to-end 처리 → 대신 특화 도구 3종 + LLM 조율자.
- ⚠️ **오픈모델의 도구호출 신뢰성**: Qwen3-8B는 개발엔 충분하나 다단계 오케스트레이션에서 도구 포맷 이탈 발생 가능 → 프로덕션은 **Qwen3-30B-A3B**로 승급, 도구인자 스키마 검증·재프롬프트, 도구선택 턴 `/no_think`.
- ⚠️ **로컬 GPU 제약(RTX 3060 12GB)**: bf16 미적재 → 4-bit AWQ 필수, FP8 미지원(Ampere), 30B-A3B 로컬 불가, 백본+VLM 동시상주 불가(VLM은 클라우드 분리). 상세 §4.5-0.6.

**운영 리스크:**
- OpenSky 크레딧 정책 변동 → 착수 전 재확인, adsb.fi 이중화.
- FIRMS `MAP_KEY` 5000요청/10분 한도 → 백엔드 캐시·bbox 최소화, 키 프론트 노출 절대 금지.
- TLE 오차 누적 → 일일 재수집 파이프라인 필수.
- deck.gl 자체 `GlobeView`는 실험적(pitch/bearing 제한) → **MapLibre globe 위 오버레이** 방식 채택으로 회피.
- **`MapboxOverlay`+globe interleaved 오클루전 파리티**(궤도가 지구 뒤로 정확히 가려지는지) → visgl/deck.gl #7920. **P0에서 최우선 스파이크 검증**.
- **Three.js globe custom layer**: 공식 지원되나 Mercator 경로보다 최신·생태계 얇음. threebox는 globe 미완성 → raw Three.js + `getMatrixForModel`, 실축척 고도 depth/scale 튜닝 시간 확보. ⚠️ 타입명 `CustomRenderMethodInput`은 typedoc으로 재확인.
- **Qwen3 vLLM reasoning+tool 파싱 버그**(#19051,#19513) → thinking off 또는 파서 조합 사전 검증.
- **`deepagents` pre-1.0 API 변동** → 버전 고정 + reference docs 대조.
- MapLibre globe 줌~12 Mercator 폴백/subdivision 128 → 아크 다중 샘플링, 실축척 3D는 전용 뷰 2트랙 옵션(§4.6-B).
