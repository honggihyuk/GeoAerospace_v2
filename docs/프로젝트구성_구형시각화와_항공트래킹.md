## 0. 먼저 — 방향과 현재 구현의 간극

정직하게 짚고 시작한다. 제시하신 방향과 실제 코드 사이에 **세 군데 실질적 간극**이 있다.

| 방향 | 현재 실태 | 간극 |
|---|---|---|
| MapLibre 웹 환경 | ✅ MapLibre GL JS v5 + deck.gl + Three.js | 없음 |
| **LangChain deepagents** | ❌ **TypeScript 규칙 기반 의도 해석** + Ollama 직접 호출 | **큼** — Python 에이전트 계층이 아예 없다 |
| **궤도 데이터 RAG** | ⚠️ RAG는 있으나 코퍼스가 **손으로 쓴 10여 개 청크**([lib/knowledge.ts](../lib/knowledge.ts), 77행) | **큼** — "궤도 데이터를 많이 끌고 오는" RAG가 아니다 |
| **VLM 궤도 해석** | ⚠️ VLM 연결은 있으나 **범용 모델이라 원격탐사 해석 실패** | **큼** — GeoChat 미도입 |
| 구형 시각화 | ✅ 3중 구현 (§2) | 없음 |
| 항공 트래킹 | ✅ ADS-B 실시간 (§3) | 없음 |

**즉 시각화·데이터 수집 계층은 상당히 완성됐고, 지능 계층(에이전트·RAG·VLM)이 방향과 어긋나 있다.** 이 문서는 §2~3에서 완성된 부분을, §4~6에서 전환 경로를 다룬다.

## 2. 요구사항 ① — 위성 궤도 데이터를 구형으로 시각화

궤도를 구(球)에 올리는 방법은 하나가 아니다. 이 프로젝트는 **세 층위로 나누어** 구현했고, 각각 푸는 문제가 다르다.

### 2.1 세 가지 구형 표현

| 층위 | 기술 | 좌표계 | 푸는 문제 |
|---|---|---|---|
| **A. MapLibre 글로브** | MapLibre GL JS v5 `projection: { type: "globe" }` | 지구고정(경위도) | 2D 지도를 그대로 구에 감기 — 지상궤적·항공기·산불을 구 위에서 본다 |
| **B. Three.js 관성계 뷰** | 커스텀 `SpaceView` (WebGL) | **ECI(관성계)** | 궤도 타원을 **실축척**으로 공간에 띄운다 |
| **C. MapLibre 위 3D 레이어** | Three.js custom layer (`orbital-3d`) | 지구고정 | 2D 지도 위에 위성 본체·센서 콘을 3D로 얹는다 |

**왜 셋이 필요한가.** 지구고정 좌표에 궤도를 그리면 지구 자전 때문에 궤도가 매 순간 휘어 보인다(지상궤적은 그래서 사인 곡선이 된다). 반대로 관성계에 그리면 궤도는 **고정된 타원**이고 지구가 그 안에서 돈다 — 이것이 물리적으로 옳은 그림이다. 두 그림 모두 필요하므로 뷰를 나눴다.

### 2.2 B층 — 관성계 구형 뷰의 구현 원리

가장 핵심적인 부분이다. [components/SpaceView.tsx](../components/SpaceView.tsx)

**① 좌표 규약**
```
1 scene unit = 1000 km,  지구 반경 R = 6.371 unit
ECI(Z-up, km) → Three.js(Y-up, unit):  (x, z, −y) × 0.001
```

**② 지구를 돌리고 궤도는 고정한다**
```ts
earth.rotation.y = satellite.gstime(now);   // GMST 자전
// 궤도 링은 ECI 좌표 그대로 → 회전시키지 않음
```
이 한 줄이 "관성계 뷰"의 전부다. 궤도가 공간에 못 박혀 있고 지구가 그 안에서 자전한다.

**③ 궤도 타원 생성**
한 주기를 256등분해 SGP4로 전파하고 ECI 좌표를 그대로 이어 `THREE.Line`으로 그린다. 지구고정 변환을 거치지 않으므로 타원이 닫힌다.

**④ 지구 표면 — 셰이더 합성**
단순 텍스처가 아니라 프래그먼트 셰이더에서 다섯 겹을 합성한다:
```
당일 트루컬러 → (결손 시) 전날 → (결손 시) 정적 BlueMarble   … 알베도
+ 야간 도시광 (밤면)
+ 태양각 기반 터미네이터 + 박명 띠
+ GK2A 관측 격자 오버레이 (한반도)
```
등장방형(equirectangular) UV라 `lon = u×360−180`, `lat = v×180−90`으로 경위도를 역산할 수 있고, 이 덕분에 임의 영역 오버레이를 셰이더에서 직접 합성한다.

**⑤ 대기**
지구보다 3.5% 큰 구를 `BackSide` + 가산 블렌딩으로 그려 림 글로우를 만든다. Rayleigh(청)·Mie(주황)를 태양각으로 가중 — **물리 산란 적분이 아닌 경험적 근사**다.

### 2.3 위성 본체 — 스케일이라는 근본 문제

**실축척으로는 보이지 않는다.**

| | 크기 |
|---|---|
| ISS 실제 | 109 m = **0.000109 unit** |
| 화면상 | 서브픽셀 (보이지 않음) |
| 기존 구 마커 | 0.12 unit = 120 km — **이미 1,100배 과장** |

이 배율에서 위성은 화면상 10~30 px이라 고밀도 glTF의 디테일은 대부분 버려진다. 그래서 **절차적 모델**(트러스·모듈·태양전지판)로 실루엣만 만들고, GLTFLoader 연결부는 남겨 실제 모델로 교체 가능하게 했다.

**자세 정렬(LVLH).** 구형 뷰에서 위성이 "제대로 놓여" 보이려면 자세가 필요하다:
```
z = −r̂        (천저 — 지구를 향함)
y = −ĥ,  h = r × v   (궤도면 법선의 반대)
x = y × z     (≈ 비행 방향)
```
태양전지판은 모델 Y축으로 회전해 판 법선이 태양을 향한다.

**마커 LOD.** 모델만 두면 넓은 뷰에서 식(eclipse) 중인 10 px 모델이 어두운 지구에 묻혀 **추적 기능을 잃는다**. 화면 크기 고정 글로우 마커를 병행한다(멀면 점, 가까우면 모델).

### 2.4 위성 시점(POV) — 구형 뷰의 확장

「위성을 클릭하면 그 위성이 보는 화면」이 성립한다. 정지궤도 GK2A(35,780 km)에서 지구 각반경은 **8.7°** — 45° 화각에 원반 전체가 담긴다.

구현 시 걸린 함정 둘:
- **자기 센서 콘 내부를 보게 된다** — 위성→지구 원뿔을 축 방향에서 보면 화면 전체가 콘으로 덮인다
- **자기 본체가 시야를 가린다** — 카메라 코앞의 모델이 지구를 막는다

→ POV에서는 자기 모델과 자기 콘을 숨긴다.

### 2.5 구형 위 관측 격자 — LCC 재투영

기상청 GK2A 격자는 **Lambert Conformal Conic**이고 구는 **등장방형** UV다. 두 좌표계가 달라 그대로 얹으면 한반도가 엉뚱한 곳에 찍힌다.

**역방향 매핑**으로 리샘플링한다 — 격자를 출력으로 "밀어내면"(forward) 구멍과 겹침이 생기므로, 출력 픽셀마다 경위도를 구해 격자에서 값을 "당겨온다"(inverse). [lib/gk2aRaster.ts](../lib/gk2aRaster.ts)

검증: 격자에 표식을 찍고 재투영 후 원래 경위도에 나타나는지 — 서울·부산·제주·강릉 모두 **0.1° 이내**.

## 3. 요구사항 ② — 항공 트래킹

### 3.1 데이터 소스

**전용 항공 라이브러리를 쓰지 않았다.** ADS-B 공개 API를 직접 호출한다:

| 소스 | 비고 |
|---|---|
| `api.adsb.lol` | 1차 |
| `api.airplanes.live` | 폴백 |

이유: 항공 추적 "라이브러리"는 대부분 데이터 접근 래퍼이고, 여기서 필요한 것은 **좌표·고도·방위·속도** 네 값뿐이다. 라이브러리를 끼우면 의존성만 늘고 폴백 제어가 어려워진다. SSRF 가드([lib/server/safeFetch.ts](../lib/server/safeFetch.ts))로 호스트를 화이트리스트 관리한다.

### 3.2 실시간성 — dead-reckoning 보간

**문제.** ADS-B 폴링은 12초 간격인데 항공기는 250 m/s로 움직인다. 폴링 간격 동안 3 km를 이동하므로 마커가 뚝뚝 끊긴다.

**해법.** 마지막 스냅숏의 위치·방위·대지속도로 **매 프레임 추측항법(dead reckoning)** 한다:
```
경과시간 × 대지속도 × (방위 방향) → 현재 추정 위치
```
[lib/aircraft.ts](../lib/aircraft.ts)의 `deadReckon()`. 30 fps로 부드럽게 움직이고, 다음 폴링에서 실측으로 보정된다.

### 3.3 렌더링 — 수천 대를 60 fps로

**deck.gl `IconLayer`** (MapLibre `MapboxOverlay`, `interleaved: true`)를 쓴다.

| 기법 | 근거 |
|---|---|
| GPU 인스턴싱 | 마커 수천 개를 draw call 한 번으로 |
| `getAngle: −track` | 기수 방위로 아이콘 회전 |
| 카테고리별 색상 | commercial / private / jet / mil |
| `useRef` + 수동 리렌더 | 수천 엔티티를 React state에 넣지 않는다 (OSIRIS 차용) |
| `document.hidden` 가드 | 백그라운드 탭에서 폴링 중단 |

**실측:** 현재 화면 기준 **1,992대** 동시 표시.

### 3.4 항공기 아이콘

런타임에 캔버스로 생성한다([`makePlaneIcon`](../lib/aircraft.ts)) — 외부 이미지 자산 없이 SDF 유사 마스크로 만들어 색상을 데이터로 입힌다.

### 3.5 "궤도를 도는 항공" — 두 궤적의 통합

이 프로젝트는 **위성 궤도와 항공 항적을 같은 화면**에서 다룬다:

| | 위성 | 항공기 |
|---|---|---|
| 소스 | TLE (Space-Track) | ADS-B |
| 전파 | SGP4 | dead-reckoning |
| 갱신 | 90분 | 12초 |
| 고도 | 400~35,800 km | 0~14 km |
| 레이어 | PathLayer(궤도)·ScatterplotLayer | IconLayer |

고도 스케일이 2,500배 차이라 같은 3D 씬에 실축척으로 두면 항공기가 지표에 붙어 보인다. 현재는 **2D 글로브(A층)에서 항공기를, 관성계 뷰(B층)에서 위성을** 주로 다룬다.

## 4. 지능 계층 — 현재 실태

### 4.1 에이전트 (현재: TypeScript 규칙 기반)

[lib/agent.ts](../lib/agent.ts)의 `resolveIntent()`가 정규식으로 의도를 판정하고 도구를 실행한다.

**현재 도구 6종:** `fly_to_place` · `select_satellite` · `toggle_layer` · `filter_fires` · `add_layer` · `analyze_image`

**도구 연쇄도 구현됨** — 산불 질의 하나가 `filter_fires → add_layer → analyze_image` 3계층으로 전개된다(fire-analyst).

**왜 규칙 기반인가.** 로컬 소형 모델(Qwen3-8B)의 도구 선택 변동성이 커서, 지도 명령은 결정론적으로 처리하고 지식 질문만 LLM으로 보낸다. **좌표는 모델이 아니라 시스템이 지오코딩으로 만든다**(환각 방지).

### 4.2 RAG (현재: 소규모 사실 코퍼스)

| 항목 | 현재 |
|---|---|
| 임베딩 | `bge-m3` (Ollama) |
| 검색 | 코사인 유사도 + **어휘 부스트 하이브리드** |
| 코퍼스 | [lib/knowledge.ts](../lib/knowledge.ts) — **손으로 쓴 10여 청크** |
| 생성 | Qwen3-8B |

**방향과의 간극이 여기다.** "위성 궤도 데이터를 많이 끌고 오는 RAG"가 아니라 궤도역학 용어 Q&A 수준이다. §5에서 확장안을 다룬다.

### 4.3 VLM (현재: 연결됨, 그러나 신뢰 불가)

`qwen2.5vl:7b`(Ollama)로 GIBS 영상을 해석한다. **파이프라인은 정상**(영상을 육안 확인함) 이지만 **모델이 부적합**하다 — 7월 캘리포니아 영상을 "눈 덮인 산악"으로 해석하고 화재를 놓쳤다.

원인은 **범용 VLM에 원격탐사 도메인 그라운딩이 없다**는 것. §6이 이에 대한 답이다.

## 5. 전환안 ① — LangChain deepagents

### 5.1 조사 결과

[deepagents](https://github.com/langchain-ai/deepagents)는 LangChain이 배포하는 **에이전트 하니스**다. LangGraph 런타임 위에서 동작하며 다음이 기본 제공된다:

| 기능 | 내용 |
|---|---|
| 서브에이전트 | 내장 `task` 도구로 격리된 컨텍스트의 자식 에이전트 생성 |
| 계획 도구 | `write_todos` — pending/in_progress/completed 상태 관리 |
| 가상 파일시스템 | `ls` `read_file` `write_file` `edit_file` `glob` `grep` `execute` |
| 모델 | `provider:model` 규약 — **`ollama:` 접두사로 로컬 모델 사용 가능** |

```python
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="ollama:qwen3:8b",          # 현재 쓰는 로컬 모델 그대로
    tools=[filter_fires, get_tle, add_layer, analyze_image],
    subagents=[fire_analyst, orbit_analyst],
    system_prompt="...",
)
```

**중요한 발견:** deepagents가 `ollama:` 접두사를 지원하므로 **현재 Ollama 자산(Qwen3-8B, bge-m3, qwen2.5vl)을 그대로 재사용**할 수 있다. 모델 교체 없이 에이전트 계층만 갈아끼우면 된다.

### 5.2 전환 설계

```
[브라우저]  MapLibre + deck.gl + Three.js
     │  (지도 조작은 그대로 클라이언트가 실행 — 왕복 지연 회피)
     ▼
[Next.js API]  /api/agent  ──프록시──▶  [Python deepagents 서비스]
     │                                        │
     │  기존 데이터 라우트 유지                 ├─ 도구: get_tle, filter_fires,
     │  (/api/tle, /api/fires, /api/gk2a …)   │        add_layer, analyze_image,
     │                                        │        query_orbit_rag
     ▼                                        └─ 서브에이전트: fire-analyst,
[외부 소스] Space-Track · FIRMS · GIBS · KMA              orbit-analyst, imagery-analyst
```

**전환 원칙 셋:**
1. **지도 조작은 클라이언트에 남긴다.** 카메라 이동을 서버 왕복으로 처리하면 반응성이 무너진다. 에이전트는 *무엇을 할지*만 결정하고 실행은 클라가 한다(현재 구조 유지).
2. **결정론적 그라운딩을 버리지 않는다.** 좌표·NORAD 번호는 계속 시스템이 만든다. deepagents는 도구 *선택*을 담당한다.
3. **데이터 라우트는 그대로 둔다.** 이미 검증된 14개 API를 deepagents의 도구로 감싸기만 한다.

### 5.3 도구 매핑

| 현재 (TS) | deepagents 도구 (Python) | 데이터원 |
|---|---|---|
| `select_satellite` | `get_tle(query)` | Space-Track |
| `filter_fires` | `filter_fires(region, day_range, min_frp)` | FIRMS |
| `add_layer` | `add_layer(layer, date)` | GIBS WMTS |
| `analyze_image` | `analyze_image(bbox, question)` | GIBS + VLM |
| — (신규) | `query_orbit_rag(question)` | §5.4 |
| — (신규) | `get_gk2a(wave_type, unit, datetime)` | 기상청 |

## 6. 전환안 ② — Hugging Face

### 6.1 VLM: GeoChat

조사 결과 [**GeoChat**](https://github.com/mbzuai-oryx/geochat)(CVPR 2024)이 현재 요구에 가장 근접하다.

| 항목 | 내용 |
|---|---|
| HF 모델 | `MBZUAI/GeoChat-7B` |
| 기반 | LLaVA-1.5 + **LoRA 파인튜닝** (전체 튜닝이 아니라 대화·지시 수행 능력을 보존) |
| 학습 | 원격탐사 지시 데이터 **318k 쌍** |
| 능력 | 장면 분류, 이미지/영역 캡셔닝, VQA, **영역 지정 대화(grounded)** |

**왜 이것이 답인가.** 현재 실패의 원인이 "범용 모델에 원격탐사 그라운딩 부재"였고, GeoChat은 정확히 그 격차를 메우도록 학습됐다. 특히 **영역 지정 대화**는 "이 bbox 안의 연기 확산"처럼 우리가 필요한 질의 형태다.

**도입 경로 둘:**
- HF Inference Endpoints (관리형, GPU 과금)
- 자체 호스팅 (vLLM). 7B라 24 GB GPU에서 동작 가능

**교체 지점은 한 곳이다.** [app/api/analyze-image/route.ts](../app/api/analyze-image/route.ts)의 `VLM_MODEL`/`OLLAMA` 환경변수만 바꾸면 된다 — 이 라우트는 처음부터 모델 교체를 전제로 썼다.

### 6.2 임베딩

현재 `bge-m3`를 Ollama로 돌리고 있다. HF에도 `BAAI/bge-m3`가 있어 **같은 모델을 Inference Endpoints로 옮길 수 있다** — 코퍼스가 커지면(§5.4) 이 편이 유리하다.

## 7. 궤도 데이터 RAG 확장안

방향의 "위성 궤도 데이터를 많이 끌고 오는 RAG"를 실현하려면 코퍼스를 무엇으로 채울지가 관건이다. **주의: 궤도 데이터는 대부분 수치라 그대로 임베딩하면 검색이 되지 않는다.** 텍스트화 전략이 필요하다.

| 계층 | 내용 | 규모 감각 |
|---|---|---|
| **개념** | 궤도역학 용어·데이터소스 설명 (현재 코퍼스) | ~10 청크 |
| **카탈로그** | Space-Track SATCAT — 위성명·발사일·국가·궤도유형·소멸여부 | 수만 객체 |
| **이력 서술** | TLE 이력을 문장으로 변환: "ISS는 2026-06-10에 반장축 +1.55 km 재부스트" | 위성×사건 |
| **관측 메타** | GK2A 채널 특성, FIRMS 탐지 요약 | 수백 |
| **문서** | 설계서·제안서·API 가이드 청크 | 수백 |

**이 프로젝트가 이미 가진 강점:** [lib/maneuvers.ts](../lib/maneuvers.ts)가 TLE 이력에서 **기동을 자동 검출**한다. 이것을 문장으로 뽑으면 "이력 서술" 계층이 자동 생성된다 — 손으로 쓸 필요가 없다.

**벡터 저장소.** 현재는 인메모리다. 수만 객체 규모로 가면 pgvector 또는 HF Datasets + FAISS로 옮겨야 한다.

## 9. 권장 진행 순서

| 단계 | 내용 | 근거 |
|---|---|---|
| **1** | **GeoChat 도입** | 교체 지점이 환경변수 한 곳. 현재 VLM 실패를 직접 해결. 투자 대비 효과 최대 |
| **2** | **RAG 코퍼스 확장** — 기동 검출 결과 자동 서술화부터 | 이미 있는 자산을 텍스트로 뽑는 것이라 신규 개발이 적다 |
| **3** | **deepagents 전환** | 도구가 6개로 늘어 규칙 기반의 한계가 보이기 시작. 서브에이전트 오케스트레이션이 필요해지는 시점 |
| **4** | 항공·위성 통합 뷰 | 고도 스케일 2,500배 문제를 어떻게 다룰지 설계 필요 |

**1·2를 먼저 하는 이유:** deepagents 전환은 아키텍처 변경이라 되돌리기 어렵다. 반면 GeoChat과 RAG 확장은 기존 구조 위에 얹는 것이라 실패해도 손실이 작다. 그리고 **VLM이 신뢰 가능해져야** 에이전트가 그것을 도구로 쓸 의미가 생긴다.
