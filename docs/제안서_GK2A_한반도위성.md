# 제안서 — 천리안위성 2A호(GK2A) 기반 한반도 위성관측 고도화

> 기상청 「위성자료(경량화) 조회서비스」(`WthrSatlitInfoService`)를 축으로,
> 현 플랫폼의 **전지구·극궤도 중심 관측을 한반도·정지궤도 관측으로 보강**한다.
> 작성 2026-07-20. 근거 수치는 모두 이 문서 §1의 실측이며, 추정은 그렇다고 표시했다.

---

## 0. 왜 지금 GK2A인가 — 현 플랫폼의 구조적 한계

현재 산불 파이프라인은 **극궤도 위성**(VIIRS/MODIS)에 전적으로 의존한다. 여기서 오는 한계는 데이터가 아니라 **궤도 기하**에서 온다:

| | 현재 (VIIRS/MODIS) | GK2A |
|---|---|---|
| 궤도 | 극궤도 — 같은 지점 **하루 2~4회** 통과 | **정지궤도**(128.2°E) — 항상 응시 |
| 관측 주기 | 통과 시각에 종속 | **2분 간격** |
| 야간 | 통과 시각에만 | 동일하게 2분 간격 |
| 커버리지 | 전지구 | 한반도(남한) |

즉 현재 구조로는 **"불이 언제 시작해서 어느 방향으로 얼마나 빨리 번졌는가"를 원리적으로 답할 수 없다.** 하루 두 장의 스냅숏뿐이기 때문이다. 이 결손은 모델이나 알고리즘으로 메울 수 없고, 관측 주기가 다른 센서로만 메워진다.

### 오늘 실측으로 확인된 또 하나의 근거

FIRMS+GIBS+VLM 3계층을 구현해 캘리포니아 산불에 적용한 결과, **VLM 해석 단계가 실질적으로 실패**했다(§4.3 상세). 원인 중 하나가 **단일 프레임**이라는 점이다 — 정지 영상 한 장에서는 연기와 구름·설원을 구분하기 어렵다. GK2A의 2분 간격은 이 문제에 직접 대응한다: **움직이는 것이 연기, 움직이지 않는 것이 지형**이다.

## 1. GK2A API 실측 사양

*가이드 문서(48p) 정독 + 실제 호출로 확인한 값. 추정치 없음.*

### 1.1 서비스 개요

| 항목 | 값 |
|---|---|
| 서비스 | `WthrSatlitInfoService` (기상청 1360000) |
| 엔드포인트 | `apis.data.go.kr/1360000/WthrSatlitInfoService` |
| 프로토콜 | REST GET, XML/JSON. **HTTPS 동작 확인** (가이드는 http로 표기하나 https도 응답) |
| 인증 | `serviceKey` (공공데이터포털 발급) — 미인증 시 `401 Unauthorized` |
| 처리량 | **30 TPS** |
| 최대 응답 | 한반도 조회 **1,126,400 byte** / 행정구역 조회 1,024 byte |

### 1.2 상세기능 10종 = 5채널 × 2범위

| 채널 | 한반도 | 행정구역 | 화재 관련성 |
|---|---|---|---|
| 적외 IR | `getGk2aIrAll` | `getGk2aIrArea` | 지표온도·운정온도 |
| 근적외 NR | `getGk2aNrAll` | `getGk2aNrArea` | 식생·설빙 구분 |
| **단파적외 SW** | `getGk2aSwAll` | `getGk2aSwArea` | **화재 탐지 핵심 채널(3.8μm대)** |
| 가시 VI | `getGk2aViAll` | `getGk2aViArea` | 연기 플룸(주간 한정) |
| 수증기 WV | `getGk2aWvAll` | `getGk2aWvArea` | 대기 흐름·수송 |

- 한반도 조회: 격자 배열 반환 (`xdim`×`ydim`, 예 320×396, `gridKm` 2.0)
- 행정구역 조회: `dongCode`로 **단일 지점 값** — 응답 1 KB, 평균 100 ms

### 1.3 좌표계 (중요 — 여기서 틀리면 전부 어긋난다)

```
투영     Lambert Conformal Conic
지구반경  6371.00877 km
표준위도  30° / 60°
기준경도  126°,  기준위도 38°
격자간격  0.5 / 1 / 2.0 km
기준점 X,Y  응답 메시지의 x0, y0 참조 (변경될 수 있음)
```

현 플랫폼은 전부 WGS84 경위도 기반이므로 **LCC ↔ WGS84 변환 계층이 반드시 필요**하다. 가이드 39~47p에 C 예제가 있으나 TypeScript 포팅이 필요하다. *(A2에서 IAU 프레임 변환을 구현하며 검증한 방식 — 기준 구현과 대조하는 테스트 — 을 그대로 적용할 것.)*

### 1.4 제약 — 착수 전 반드시 인지할 것

| 제약 | 내용 | 영향 |
|---|---|---|
| **관측 지연** | **2일 전 ~ 6시간 전**만 조회 가능 | ❗ **"실시간"이 아니다.** 실시간 경보용으로 쓸 수 없다 |
| **커버리지** | **남한만** 제공 (북한·국외 미제공) | 한반도 전체 분석 불가 |
| **값 형식** | Count 번호로 반환 (예: 1562) | 변환테이블 필수 — Radiance/Albedo/BT로 환산 |
| 변환테이블 | `20190415_GK-2A_AMI_Conversion_Table_v3.0.zip` (**3.7 MB, 다운로드 확인**) | 서버에 내장하거나 기동 시 로드 |
| 인증 | `serviceKey` 필요 | 사용자가 공공데이터포털에서 발급 |

> **가장 중요한 제약은 6시간 지연이다.** 이 문서의 어떤 제안도 "실시간 산불 경보"를 표방하지 않는다. GK2A가 주는 것은 **실시간성이 아니라 시간 해상도**다 — 이미 일어난 일을 2분 간격으로 되짚는 능력.

## 2. 제안 항목

### K1. LCC 좌표 변환 + 격자 파이프라인 — *전제, 낮은 노력* ★★★

모든 후속 작업의 토대. 단독으로는 사용자 가치가 없지만 없으면 아무것도 못 한다.

- `lib/lcc.ts`: LCC ↔ WGS84 양방향 변환 (가이드 C 예제 포팅)
- Count → 물리량 변환 테이블 로더
- 1차원 배열 → 2차원 격자 복원 (가이드 37p 변환코드)
- **검증**: 가이드가 제시한 기준값 `X=59, Y=125 ↔ 126.929810°E, 37.488201°N` 을 테스트로 고정. 왕복 변환 오차도 함께 검사.

### K2. 한반도 화재 2분 간격 시계열 — *핵심 가치* ★★★

**현 플랫폼이 원리적으로 못 하던 것을 가능하게 하는 항목.**

- `getGk2aSwAll`(단파적외)를 2분 간격으로 수집 → **화재 성장 애니메이션**
- 기존 타임 컨트롤러(B2)에 그대로 연결 — 배속/스크럽 UI가 이미 있다
- FIRMS 탐지점(현재 구현)과 **교차 검증**: FIRMS가 극궤도 통과 시각에 찍은 점이, GK2A 시계열의 어느 시점에 나타나고 사라지는지
- 산출: "발화 추정 시각", "확산 속도(km/h)", "피크 시각"

> 이것이 GK2A를 도입하는 **주된 이유**다. 나머지 항목은 이것의 보조이거나 파생이다.

### K3. VLM 시계열 해석 — *오늘의 실패에 대한 직접 대응* ★★

§4.3에서 확인했듯 단일 프레임 VLM 해석은 실패했다. GK2A는 이 문제에 구조적으로 대응한다.

- 연속 프레임 3~5장을 **한 장의 격자 이미지로 합성**해 VLM에 제시 → "무엇이 움직였는가"를 묻는다
- 움직이는 밝은 영역 = 연기/구름, 고정된 고온 영역 = 화재
- 기대: 정지 영상에서 실패한 "연기 vs 설원" 구분이 시간 축으로 해결됨
- **단, GeoChat 등 원격탐사 특화 VLM 확보가 병행되어야 한다** (§4.3)

### K4. 행정구역 단위 질의 — *에이전트 친화, 낮은 노력* ★★

`getGk2a*Area`는 `dongCode`로 **단일 지점 값을 1 KB / 100 ms**에 준다. 격자 전체(1.1 MB)를 받을 필요가 없다.

- 에이전트 도구 `query_kma_point(region, channel, time)` 
- "지금 대구 상공 운정온도는?" 같은 질의를 지오코딩 → dongCode → 즉답
- 현 플랫폼의 지오코딩(§4.3 환각 방지 원칙)에 dongCode 매핑을 추가

### K5. 데이터 신선도 정직 표기 — *플랫폼 철학의 연장* ★★

이 프로젝트는 A2/A4에서 **추정치를 실측치로 교체**하고, TLE 소스가 일부라도 데모면 `LIVE` 대신 `PARTIAL`로 표기하는 원칙을 세웠다. GK2A 도입 시 소스별 지연이 크게 달라지므로 같은 원칙을 적용한다:

| 소스 | 지연 |
|---|---|
| GK2A (이 API) | **6시간~2일** |
| FIRMS NRT | 통과 후 ~3시간 |
| GIBS 트루컬러 | 당일분이 부분 수집 (B1 실측) |

→ 각 레이어에 **관측 시각과 지연**을 명시. 서로 다른 시각의 데이터를 겹쳐 보여주면서 같은 순간인 척하지 않는다.

### K6. GK2A 자체를 궤도 뷰에 — *낮은 노력, 서사적 연결* ★

플랫폼이 이미 위성을 추적한다. GK2A(NORAD 43823)를 추적 대상에 추가하고, 3D 뷰에서 **정지궤도 위치 + 한반도 관측 풋프린트**를 표시한다. "이 데이터를 만드는 위성이 저기 저것"이라는 연결은 다른 어떤 플랫폼도 쉽게 못 하는 우리 고유의 장면이다.
*(B4의 풋프린트 계산이 이미 구현돼 있어 재사용 가능. 단 정지궤도는 λ≈81°로 값이 매우 커진다 — B4 테스트에 GEO 케이스가 이미 포함돼 있다.)*

## 3. 우선순위

| 항목 | 효과 | 노력 | 우선 | 비고 |
|---|---|---|---|---|
| K1 LCC 변환·격자 | 전제 | 낮음 | ★★★ | 없으면 아무것도 안 됨 |
| K2 2분 간격 화재 시계열 | **원리적 신규 역량** | 중 | ★★★ | 도입의 주된 이유 |
| K5 신선도 정직 표기 | 신뢰성 | 낮음 | ★★ | 지연 편차가 커서 필수 |
| K4 행정구역 단위 질의 | 에이전트 UX | 낮음 | ★★ | 응답 1KB/100ms |
| K3 VLM 시계열 해석 | 해석 품질 | 중 | ★★ | GeoChat 확보 선행 |
| K6 GK2A 궤도 표시 | 서사 | 낮음 | ★ | 기존 B4 재사용 |

**권장 착수 순서: K1 → K2 → K5**. 이 셋이 "한반도를 2분 간격으로 정직하게 되짚는다"는 가치를 완성한다. K3/K4/K6은 그 위의 선택지다.

## 4. 선행 조건과 리스크

### 4.1 사용자가 준비해야 할 것

**`serviceKey` 발급** — [공공데이터포털](https://www.data.go.kr)에서 「기상청_위성자료(경량화) 조회서비스」 활용신청.
발급 후 `.env.local`에 서버 전용으로 넣는다 (Space-Track·FIRMS와 동일 패턴):

```
KMA_SERVICE_KEY=<발급키>
```

> 채팅에 붙여넣지 마세요. 서버 전용 환경변수(`NEXT_PUBLIC_` 접두사 금지)로만 읽습니다.

### 4.2 기술 리스크

| 리스크 | 완화 |
|---|---|
| LCC 변환 오차가 조용히 전파 | 가이드 기준값(`X=59,Y=125 ↔ 126.9298,37.4882`)을 테스트로 고정. A2 프레임 변환에서 쓴 방식 |
| Count→물리량 변환 누락 | 변환 없이 쓰면 값이 무의미. 테이블 미탑재 시 **에러로 실패**시키고 조용히 진행하지 않는다 |
| 2분 간격 수집 시 30 TPS 초과 | 시계열 1시간 = 30프레임. 배치 간격 제어 + 서버 캐시 (Space-Track에서 쓴 패턴) |
| 6시간 지연을 실시간으로 오인 | K5로 UI에 지연 명시. 이 문서도 "실시간" 표현을 쓰지 않음 |
| 남한 한정 커버리지 | 경계 밖 질의는 명시적으로 "미제공"으로 응답. 빈 격자를 0으로 채우지 않는다 |

### 4.3 VLM에 대한 정직한 평가 — *오늘 실측*

P5.5에서 `analyze_image`를 구현하고 실제로 돌린 결과를 그대로 기록한다. **K3의 전제가 되는 사실이다.**

- **파이프라인은 정상**: GIBS WMS로 관심영역을 잘라 받아(768px, 130 KB) 로컬 VLM에 전달하는 경로는 정상 작동. 이미지를 육안 확인한 결과 대상 지역(Susanville/Honey Lake)의 정확한 위색합성 영상이었다.
- **VLM이 실패**: `qwen2.5vl:7b`는 7월 캘리포니아 40.6°N 영상을 **"눈 덮인 산악"** 으로 해석했고, 화재 신호를 찾지 못했다. bands721(화재가 붉게 보이는 위색합성)로 바꿔도 동일하게 실패.
- **원인**: 범용 VLM은 원격탐사 도메인 그라운딩이 없다. 밝은 지표를 설원으로, 연기를 구름으로 혼동한다. 제안서 §4.4가 **GeoChat**(원격탐사 특화)을 지목한 이유가 정확히 이것이다.
- **처리 시간**: 3~10초 (로컬 Ollama, 768px). 온디맨드 도구로는 수용 가능.

**결론:** VLM 해석 계층은 **모델을 바꾸기 전까지 신뢰할 수 없다.** K3는 GeoChat 또는 동급 원격탐사 VLM 확보를 전제로 하며, 그 전까지 VLM 답변은 참고 표기와 함께 제공해야 한다.

추가 개선안(미구현): FIRMS 탐지점을 영상 위에 **마커로 합성한 뒤** VLM에 넘기는 시각적 그라운딩. 모델에게 "어디를 보라"를 알려주는 방식으로, 모델 교체 없이도 개선 여지가 있다.

## 5. 현 플랫폼과의 접점

이 제안은 신규 구축이 아니라 **기존 자산의 재사용**이다:

| 재사용 대상 | 구현 위치 | GK2A에서의 쓰임 |
|---|---|---|
| 타임 컨트롤러 (배속·스크럽) | B2 | K2 시계열 재생 |
| 풋프린트 계산 | B4 `lib/passes.ts` | K6 정지궤도 관측 범위 |
| SSRF 가드 + 서버 캐시 | `lib/server/safeFetch.ts` | K1 API 호출 |
| 지오코딩(환각 방지) | `lib/geo.ts` | K4 지역명 → dongCode |
| 레이어 지연 로딩 | `lib/firesClient.ts` | K2 레이어 |
| 소스 정직 표기 | A1 `PARTIAL` 패턴 | K5 지연 표기 |
| 기준 구현 대조 테스트 | A2 `lib/frames.test.ts` | K1 LCC 검증 |
| VLM 해석 도구 | P5.5 `/api/analyze-image` | K3 시계열 해석 |

## 6. 요약

GK2A가 주는 것은 **실시간성이 아니라 시간 해상도**다. 6시간 지연이라는 명확한 한계가 있지만, 그 대가로 **2분 간격**이라는, 극궤도로는 원리적으로 불가능한 능력을 얻는다.

현 플랫폼의 산불 파이프라인은 하루 두 장의 스냅숏에 기반해 "지금 불이 있다"까지만 말할 수 있다. GK2A를 붙이면 **"언제 시작해 어디로 얼마나 빨리 번졌다"** 를 말할 수 있게 된다. 이것이 이 제안의 전부이며, 나머지는 그 주변이다.

착수는 **K1(변환) → K2(시계열) → K5(정직 표기)** 순으로 권장한다.
