## 2. MapLibre globe 위 Three.js custom layer 통합

- **기반**: MapLibre GL JS v5의 custom layer(`type:"custom", renderingMode:"3d"`)와 globe 투영.
- **사용 기법**: MapLibre가 넘겨주는 GL 컨텍스트를 Three가 **공유**한다
  (`new THREE.WebGLRenderer({ canvas: map.getCanvas(), context: gl, antialias:true }); renderer.autoClear=false`).
  프레임마다 `render(gl, args)`에서 `args.defaultProjectionData.mainMatrix`를 받아 카메라에 주입.
- **구현** — [`orbitalLayer.ts`](../lib/three/orbitalLayer.ts) `createOrbitalLayer()`
  - 한 레이어가 **위성 3D 모델 + 항상 보이는 마커 구 + 센서 콘 + 실축척 궤도링 + 깊이 구 +
    대한민국 큐브샛 + 한반도 큐브 그리드** 를 모두 담당(별도 씬으로 패스 분리).
  - 위성 애니메이션을 위해 render 끝에서 `map.triggerRepaint()`(연속 리페인트).
- **실습**: 런타임 스모크로 layer 마운트·콘솔 에러 0 확인. globe 미지원 폴백(guard: `getMatrixForModel` 부재 시 skip).

### 2.1 핵심 좌표 트릭 — `getMatrixForModel` translation 추출

- **문제**: globe 투영은 mercator→구면 워핑을 **셰이더에서** 하므로, 단일 4×4 행렬로는 globe 전역에
  퍼진 지오메트리(궤도링 등)를 올바르게 못 놓는다.
- **기법**: `transform.getMatrixForModel([lng,lat], alt)` 가 반환하는 모델행렬 **M의 translation 열
  `(m[12],m[13],m[14])`** 은 그 지점 원점의 월드좌표다
  (`mainMatrix · M · [0,0,0,1] = mainMatrix · Mtranslation`). 그래서 **정점마다 M의 translation만 뽑아**
  BufferGeometry에 담고 `projectionMatrix = mainMatrix` 로 그리면, **globe 곡률이 정점별로 정확히 반영된**
  선/면을 단일 draw로 얻는다. ECEF 수동 변환·접평면 선형화 오차가 없다.
- **실습 (정량 검증)**: 임시 라우트 `/mltest`(검증 후 삭제)에서
  - 실제 render-pass 안의 `mainMatrix`를 캡처해 `mainMatrix ⊗ getMatrixForModel-translation`을
    화면좌표로 변환 → MapLibre 자체 `map.project()`와 픽셀 비교.
  - **결과: 앞면 5개 지점 dpx = 0.0px**(수평 배치 완전 일치), **림 근처 고도 500km → 17.6px 반경
    바깥으로**(고도 리프트 방향·크기 정상).
  - hidden 탭이라 rAF가 멈춰 있어 `requestAnimationFrame`을 `setTimeout`으로 shim해 MapLibre 렌더를
    강제 구동한 뒤 측정(환경 우회 기법).

## 3. 깊이 전용 지구 구 (occlusion)

- **기법**: 지구 뒤편으로 넘어간 궤도·위성·큐브를 가리기 위해, **위경도 격자 구**를 `colorWrite:false,
  depthWrite:true`로 먼저 그려 지표 깊이를 기록한다(색은 MapLibre 지구 그대로, 깊이만 남김).
- **구현**: `buildSphere()`/`rebuildSphere()`. 정점은 매 rebuild 시 `getMatrixForModel([lon,lat],0)`
  translation으로 채운다. **`side: THREE.DoubleSide`** — globe 월드 공간에서 삼각형 감김이 예측 불가라
  단면이면 뒷면 컬링으로 근접 반구 깊이가 안 써져 가림이 무력화된다(양면으로 감김 무관하게).
- **성능**: 투영은 zoom morph에서만 바뀌므로 `'move'` 시(`moveDirty`)에만 재구성. 정지 애니메이션 시 재계산 0.

## 4. 궤도 표현 — 닫힌 관성계 타원(순간 타원)

- **문제 발견(실습)**: 처음엔 궤도링을 ECEF `[now, now+주기]` 로 그렸더니, 지구자전 세차로 시작·끝이
  ~20°+ 벌어진 **열린 나선**이 되어 "SpaceView(ECI)처럼 이어지지 않는다"는 피드백.
- **기법**: 궤도링 = **순간 궤도 타원**. 한 주기 ECI(관성계) 형상을 **now 하나의 GMST로만** 회전
  → 시작·끝이 만나는 닫힌 타원(= SpaceView 관성계 궤도와 같은 외형). 지상궤적(track)은 반대로
  각 점의 실제 시각 GMST(세차 반영)로 계산.
- **구현** — [`orbit.ts`](../lib/orbit.ts) `computeOrbit()`:
  `const gmst0 = gstime(now)` 고정 → `ring[i] = eciToGeodetic(propagate(t_i), gmst0)`,
  `track` 은 `eciToGeodetic(propagate(t_i), gstime(t_i))` + 자오선 분할.
- **시간 정합(§5)**: [`MapCanvas.tsx`](../components/MapCanvas.tsx) 렌더 루프가 **시뮬 시각(simClock)** 기준으로
  궤도를 주기 재계산(`RING_REFRESH_SIM_MS`). ECEF 링은 지구자전으로 세차하므로 실시간에 한 번 얼리면
  마커와 어긋난다 — 배속/스크럽/일시정지에서도 정합하려면 simClock 기준 재계산이 필수.
  링 시작점 == 마커(같은 satrec·같은 시각)라 마커가 궤도 위에 정확히 얹힌다.

## 5. 구름 없는 실사 베이스맵 (GIBS BlueMarble)

- **기법**: 2D globe 베이스맵을 CARTO 다크(양식화)에서 **NASA GIBS `BlueMarble_ShadedRelief_Bathymetry`**
  WMTS로 교체. 정적 합성이라 **구름·궤도 스와스가 없고**, 지형 음영 + 해저 수심으로 지면이 또렷.

  (`…/BlueMarble_ShadedRelief_Bathymetry/default/default/GoogleMapsCompatible_Level8/{z}/{y}/{x}.jpeg`,
  `maxzoom:8`). DEM(terrain)에 드레이프되어 3D 기복도 함께. 실사 트루컬러는 GIBS 오버레이로 선택.

## 6. 16U 큐브샛 — glTF(DRACO) 모델

### 6.1 로드 파이프라인 — [`orbitalLayer.ts`](../lib/three/orbitalLayer.ts) `onAdd()`
- **기반**: 사용자 제공 `CubeSat - 2 RU Generic.glb`(206 KB) → `public/models/cubesat.glb`.
```ts
const gltfLoader = new GLTFLoader();
const dracoLoader = new DRACOLoader();
dracoLoader.setDecoderPath("/draco/gltf/");    // 로컬 호스팅 디코더
gltfLoader.setDRACOLoader(dracoLoader);
gltfLoader.load("/models/cubesat.glb", (gltf) => {
  const model = gltf.scene;
  const box = new THREE.Box3().setFromObject(model);
  const s = 90_000 / (box.getSize(new THREE.Vector3()).length() || 1); // 대각 ~90 km 로 과장(서브픽셀 방지)
  model.scale.setScalar(s);
  model.position.sub(box.getCenter(new THREE.Vector3()).multiplyScalar(s)); // 원점 중심 정렬
  cubeSat.add(model);
}, undefined, (err) => { console.warn("[cubesat] glb 로드 실패 → 절차적 폴백:", err); cubeSat.add(buildCubeSat()); });
```
- **배치·렌더**: `cubeSat`(빈 그룹, 스핀 담당)을 `cubeScene`(자체 3점 조명)에 넣고, render 패스에서
  `camera.projectionMatrix = mainMatrix ⊗ getMatrixForModel([127.8,36.5], 550_000)` 으로 한반도 상공 550 km에
  투영. 매 프레임 `cubeSat.rotation.z += 0.005`(완만한 자전). 깊이 구(§3) 공유로 지구 뒤편 시 자동 가림.

### 6.2 실습 — "왜 glb가 안 뜨고 절차적 모델이 떴나"
1. **증상**: glTF 대신 폴백(`buildCubeSat()`) 렌더 = `load()`의 **error 콜백이 탔다**.
2. **진단**: 브라우저에서 glb를 직접 fetch해 **glTF-binary 헤더 파싱** — magic `glTF`(0x46546C67) 확인 후
   오프셋 20의 JSON 청크를 디코드하니:
   ```json
   { "extensionsRequired": ["KHR_draco_mesh_compression"] }   // ← DRACO 압축
   ```
   GLTFLoader 단독으로는 DRACO 지오메트리를 못 풀어 **조용히 실패** → 폴백이 뜬 것.
3. **해결**: three 동봉 디코더(`node_modules/three/examples/jsm/libs/draco/`)를 `public/draco/`로 복사,
   `DRACOLoader.setDecoderPath("/draco/gltf/")` 배선. **디코더 서빙·MIME 검증**(둘 다 필수):
   - `/draco/gltf/draco_wasm_wrapper.js` → 200, `application/javascript`
   - `/draco/gltf/draco_decoder.wasm` → 200, **`application/wasm`**(WASM `instantiate`에 이 MIME가 필요)
- **마커 크기 실습**: 초기 마커가 과대(선택 시 ~42 px)로 보여 `markerBase` 260 km → **85 km** 로 축소
  (globe 줌에서 21 km/px 기준 ~8 px). §11 참조.

### 6.3 절차적 폴백 모델 (`buildCubeSat`)
glb 실패 시에도 실기체 룩을 유지하는 순수 절차 모델:
- **태양전지 셀 텍스처(캔버스, `makeSolarTexture`)**: 금박 기판 위 4×8 다크블루 **모노셀** — 모서리 챔퍼(금박
  삼각형) + 은색 버스바, `SRGBColorSpace`. 본체 전면·전개 날개에 `map`으로 입힘.
- 금박 상/하 캡 + 세로 프레임 스트립, 알루미늄 코너 레일 4, 상단 데크(카메라 배럴+렌즈, 스타트래커 2, 위프
  안테나 2), 전개형 태양 날개 4(양측 × 상/하 2단, 양면 셀), 하향 통신 안테나.

## 7. 한반도 큐브 관측 (voxel grid)

> 큐브샛 더블클릭 = "관측 태스킹". 한반도가 `GRID_NX×GRID_NY` 격자 셀(=큐브 단위)로 세분되어, 각 육지 셀이
> DEM 고도만큼 돋은 3D 복셀이 되고, 표면 색은 우상단 토글(정사영상/고도/SAR)로 바뀐다.

### 7.1 상호작용 흐름 — [`MapCanvas.tsx`](../components/MapCanvas.tsx)
큐브샛 지상 투영에 근접(≤70 px)한 **더블클릭** → `e.preventDefault()`(기본 줌 차단) → `store.cube.active=true`
→ `map.flyTo({center: KOREA_CENTER, zoom: 4.6, pitch: 45})` → `map.once("idle", …)`(DEM 타일 로드 대기) →
`observeKorea()` + `applySurface()`. 다시 더블클릭 시 해제(`koreaGridRef=null`).

### 7.2 높이 — `observeKorea()`
```ts
for (each cell) heights[i] = (map.queryTerrainElevation([lng,lat]) ?? 0) / 1.25; // 지형 과장(1.25) 되돌림
heightsCache = heights;   // 표면 토글 시 재사용(재샘플 없음)
```
Copernicus DEM은 남한 미제공(부록 A)이라 **키리스 Terrarium 30m** 사용. `idle` 이후라 타일 로드 보장.

### 7.3 표면 색 — `applySurface()` (점진적 로딩)
1. **즉시 고도색**으로 그리드 표시 → SAR/정사영상 ~4초 대기 중 빈 화면 방지.
2. 선택 소스 raster fetch → 셀별 샘플 → 색 교체(**레이스 가드**: 그새 소스 바뀌거나 비활성이면 폐기):
   - `ortho` → `/api/vworld` → 실측 RGB 그대로.
   - `sar`   → `/api/sar` → 회색조 `[g, g·0.92, g·0.78]`(따뜻한 톤 → 고도색과 구분).
   - `dem`   → 1)의 고도색이 곧 최종.
   - 샘플 좌표 `sampleIdx(x,y,iw,ih)`: 이미지 y는 위가 북이라 `py = (1 − (y+0.5)/ny)·ih` 로 **뒤집는다**.

### 7.4 렌더 — `rebuildGrid()`: 프레임 축 규약 **무가정** 배치
- **실습에서 드러난 버그**: 큐브를 앵커(한반도 중심) 로컬 프레임의 X/Y/Z에 그대로 "동/북/상"이라 가정해
  배치했더니, 격자가 **얇은 동서 띠로 ~10× 압축**됐다. 원인 — `getMatrixForModel` 프레임 축이 ENU가 아니라
  "북 offset"이 고도/포어쇼트닝 축으로 새어 남–북이 화면상 뭉개진 것.
- **해결(경험적 기저)**: 앵커 역행렬 + 이웃 지점 translation **차분**으로 동/북/상 단위벡터를 실측 유도:
  ```ts
  const anchorInv = mat(getMatrixForModel(KOREA_CENTER, 0)).invert();
  const localOf = (lng,lat,alt) => vec(getMatrixForModel([lng,lat],alt).translation).applyMatrix4(anchorInv);
  const o    = localOf(cLng, cLat, 0);
  const eHat = localOf(cLng+dDeg, cLat, 0).sub(o).multiplyScalar(1/(dDeg*mLng)); // 로컬단위/미터(동)
  const nHat = localOf(cLng, cLat+dDeg, 0).sub(o).multiplyScalar(1/(dDeg*mLat)); // /미터(북)
  const uHat = localOf(cLng, cLat, 1e4 ).sub(o).multiplyScalar(1/1e4);           // /미터(상)
  // 셀마다:
  m.makeBasis(eHat·cellW, nHat·cellH, uHat·hz)
   .setPosition(o + eHat·eastM + nHat·northM + uHat·(hz/2));   // hz = max(CUBE_MIN_M, height·CUBE_EXAG)
  mesh.setMatrixAt(k, m); mesh.setColorAt(k, colors[i]);
  ```
  이 기저로 만든 `InstancedMesh` 인스턴스 행렬을, render 패스가 `mainMatrix ⊗ getMatrixForModel(KOREA_CENTER,0)`
  로 그 앵커 프레임을 소비한다. **축 규약 가정이 전혀 없어** 어떤 프레임 배열이 와도 정확. zoom
  morph(`moveDirty`)/데이터 변경 시 기저 재유도. (한반도 ~400 km 범위라 단일 앵커 접평면 오차는 무시 가능.)

### 7.5 바다 셀 숨김 (Terrarium 수심)
- **실습(임계 결정)**: 서해안 z8 Terrarium 타일을 `sharp`로 raw 디코드(`elev = R·256 + G + B/256 − 32768`):
  - min **−685 m**, **얕은 황해 −20~−60 m(픽셀 85%가 바다)**, **해안 전이대 −10~+10 m**.
- 그래서 `SEA_LEVEL_M = -8`: `heights[i] < −8` 셀은 **인스턴스 자체를 생성 안 함**(육지·섬만 큐브화, 인스턴스
  수 = 육지 셀 수) → 한반도 형상 강조. −8이라 **"여유 있게"** 해안 저지대·갯벌·간척지는 남는다.

### 7.6 튜닝 상수 — [`koreaCube.ts`](../lib/koreaCube.ts)
`GRID_NX=100, GRID_NY=120`(44×52 → 80×96 → 100×120 로 단계 상향), `CUBE_EXAG=30`, `CUBE_GAP=0.9`(간격),
`CUBE_MIN_M=700`, `SEA_LEVEL_M=-8`, `KOREA_BBOX=(125.5, 33.9, 129.7, 38.7)`. 값만 바꿔 즉시 재튜닝.

## 8. Sentinel-1 SAR 프록시 (`/api/sar`)

### 8.1 파이프라인 — [`app/api/sar/route.ts`](../app/api/sar/route.ts)
- **기반**: Copernicus Data Space Ecosystem(CDSE) / Sentinel Hub Process API. Sentinel-1 GRD(IW) VV 후방산란.
- **OAuth `client_credentials`**(토큰 60초 여유 캐시):
  `POST identity.dataspace.copernicus.eu/auth/realms/CDSE/protocol/openid-connect/token`
- **Process 요청**(`sh.dataspace.copernicus.eu/api/v1/process`, 요지):
  ```jsonc
  {
    "input": {
      "bounds": { "bbox": [w,s,e,n], "properties": { "crs": "…EPSG/0/4326" } },
      "data": [{ "type": "sentinel-1-grd",
                 "dataFilter": { "timeRange": { …최근 45일… }, "acquisitionMode": "IW" },
                 "processing": { "backCoeff": "SIGMA0_ELLIPSOID" } }]   // 지형보정 없음(빠름·안정)
    },
    "output": { "width": w, "height": h, "responses": [{ "format": { "type": "image/png" } }] },
    "evalscript": "…"
  }
  ```
- **evalscript**(VV 선형 → dB → 0..255 UINT8):
  ```js
  //VERSION=3
  function setup(){ return { input:["VV"], output:{ bands:1, sampleType:"UINT8" } }; }
  function evaluatePixel(s){
    var db = 10.0*Math.log(Math.max(s.VV,1e-6))/Math.LN10;      // 대략 −22~0 dB
    return [ Math.round(Math.max(0,Math.min(1,(db+22)/22))*255) ];
  }
  ```
- 자격증명은 서버 env(`SENTINEL_HUB_CLIENT_ID/SECRET`). 미설정 **501**(클라 고도색 폴백), 처리 실패 **502**
  (detail 포함). `h`는 bbox 종횡비로 산출.

### 8.2 실습 — 3연쇄 디버깅 (라이브 엔드포인트)
CDSE가 **인증 통과 후** 낸 실제 에러 메시지로 순차 해결(= 키·OAuth는 처음부터 정상):
1. **해상도 하한**: `process 400` — *"Your request of 3656.75 meters per pixel exceeds the limit 1500.00
   meters per pixel of the collection S1GRD."* w=128(코스)이 원인. **w=512(≈911 m/px)** 로 통과.
2. **연결 드롭/서버 다운**: w=512에서 "Failed to fetch"(전 라우트가 무응답 = dev 프로세스 다운). 원 라우트의
   `orthorectify + GAMMA0_TERRAIN`(지형보정)이 느려 프로세스를 죽인 정황 → **`SIGMA0_ELLIPSOID`**(정사보정
   제거)로 **~4초 안정**. (사용자 dev 서버 자체가 죽어 있던 것도 함께 확인 → 재시작.)
3. **스트림 패스스루 불안정**: 성공(200) 경로의 `new NextResponse(r.body)`가 Next dev에서 불안정 →
   `const buf = await r.arrayBuffer(); return new NextResponse(buf, {…})` 로 교체.

### 8.3 검증 — 응답 픽셀 통계
w=512 PNG(512×585, ~4 s): **min 0 / max 255 / 평균 116 / 89% 유효**,
**도심(서울권) 211(밝음)** · 바다 107 · 산지 160 → SAR 후방산란 특성(도심·구조물 강산란, 잔잔한 수면 약산란)
정확. UX: 점진적 로딩(고도색 → SAR), 회색조 팔레트로 고도색과 명확 구분.

## 9. VWorld 정사영상 프록시 (`/api/vworld`)

### 9.1 파이프라인 — [`app/api/vworld/route.ts`](../app/api/vworld/route.ts)
- **기반**: VWorld(국토정보플랫폼, 국토부) Satellite WMTS(항공 정사영상, 무료 API 키 + 도메인 등록).
  VWorld **정밀 DEM API는 보안상 중단**(부록 A)이라 **정사영상 텍스처만** 승급.
- **서버 스티치**(클라 per-cell 샘플용 단일 raster 생성):
  1. bbox(경위도) → **웹메르카토르 타일 범위(z8)** 산출:
     `x = (lng+180)/360 · 2^z`,  `y = (1 − ln(tan φ + sec φ)/π)/2 · 2^z`.
  2. 각 타일 `…/req/wmts/1.0.0/{KEY}/Satellite/{z}/{y}/{x}.jpeg` fetch — **`Referer: VWORLD_DOMAIN`** 부착
     (키의 도메인 제한 통과). 매직바이트(JPEG `ffd8`/PNG `8950`)로 유효성 판별, 결측 타일 스킵.
  3. `sharp({create: 배경}).composite(tiles).png().toBuffer()` → `sharp(stitched).extract(bbox 픽셀).resize(outW,outH).png()`.
- 키는 서버 env(`VWORLD_KEY`, `VWORLD_DOMAIN`). 미설정 **501** → 클라 고도색 폴백.

### 9.2 실습 — sharp 버그 규명
1. **증상**: 502 `"Input buffer contains unsupported image format"`(sharp) → 정사영상이 계속 고도색으로 폴백.
2. **타일 직접 검증**(서버에서 `.env.local` 키로 단일 타일 fetch): status **200, valid JPEG 26 KB, head
   `ffd8ffe0`, https/http·Referer 모두 정상** → **키·타일·도메인 문제 아님**.
3. **진짜 원인**: 중간 스티치 버퍼 `sharp({create}).composite(tiles).toBuffer()`에 **포맷 지정(`.png()`) 누락**
   → raw 픽셀이 나와 다음 `sharp(stitched)`가 못 읽음. **`.composite(tiles).png().toBuffer()`** 로 해결.
4. **검증**: 정사영상 실측 RGB — 산림 [61,109,72] 녹색, 바다 [58,81,112] 파랑, 서울권 [63,94,52] → 실사 정상.

## 11. 위성 마커·클릭 피킹 — pitch 정합

- **문제 발견(실습)**: 지도를 **기울이면(pitch)** deck.gl 위성 마커(노란 점)가 궤도/모델과 어긋나
  불규칙하게 움직였다. 원인: **deck.gl이 MapLibre globe + pitch에서 고도(z) 투영을 제대로 못 맞춤**
  (getMatrixForModel로 정확히 그려지는 Three 모델/콘과 벌어짐).
- **해결**:
  - deck.gl 위성 ScatterplotLayer 제거.
  - 위성 마커를 **Three 레이어의 정확 위치**에 무광 구(sphere)로 렌더(대략 화면 고정 크기
    `markerBase=85_000/2^(zoom-2)`, 색=위성색, 선택 시 확대).
  - 클릭 선택도 render 패스에서 계산한 **정확 화면좌표**(`satHitsOut`)로 최근접 피킹 →
    pitch 무관하게 마커=모델=궤도=클릭 완전 일치.
- **부수 수정**: 센서 콘 `scale`을 `≤2.5`로 캡(고궤도에서 콘이 무한정 커지던 오류). 지상궤적 레이어 제거.

## 12. Cesium 단일엔진 이주 — 시도 후 롤백 (의사결정 기록)

- **배경**: "단일 엔진 통일" 요구로 CesiumJS 1.143 이주를 진행.
  - 설치·Next 설정(정적 자산 `public/cesium` 복사, `CESIUM_BASE_URL`), `CesiumView` 구현:
    GIBS `UrlTemplateImageryProvider`, 내장 조명/대기, **`Clock`+`SampledPositionProperty`+`PathGraphics`**
    로 시간동적 위성/궤도(궤도-마커 정합이 자동), ICRF 개념.
  - 런타임 스모크로 Viewer 마운트·에러 0 확인.
- **롤백 이유**: 사용자 요구의 본질이 **"MapLibre 2D globe 위에 닫힌 관성계 궤도"** 였고, 그건 MapLibre로
  달성 가능했다. Cesium 관련 전부 제거(파일·의존성) 후 MapLibre 경로로 복귀.
- **남긴 교훈(문서화 가치)**: 실축척 GEO/관성계 우주 시뮬이 핵심이면 Cesium이 정석이나, **닫힌 궤도
  외형**은 MapLibre에서 "순간 궤도 타원(§4)"으로 충분히 재현된다.
