## 1. 개요

기존 GeoAerospace(위성·교통 실시간)에 **Sentinel-2 위성영상 기반 지구관측 분석**을 추가했다. AWS 참조 프로젝트와 **설계 철학이 동일**하다:

> **계산은 전부 결정론적 도구가 하고, LLM은 도구가 반환한 값을 서술만 한다.**

이 철학은 Qwen3-8B 같은 소형 로컬 모델에 특히 유리하다 — 어려운 건 서술이 아니라 도구 선택인데, 그건 `resolveIntent`(정규식 결정론 라우팅)가 대신한다.

**AWS와 다른 점**: AWS는 Python(rasterio/GDAL/Bedrock/TiTiler/Lambda)이지만 우리는 **순수 Node/TypeScript**다. COG는 `geotiff.js`의 HTTP Range로 AOI 창만 읽고, 렌더는 `sharp` PNG, LLM은 Ollama, 벡터는 pgvector. **인프라(Lambda·TiTiler·AgentCore) 없이** 같은 기능을 구현했다.

**AWS 원본에서 발견해 교정한 버그가 여럿**이다(§7). "이식"이 아니라 "검증하며 재구현"에 가까웠다.

## 3. 분광지수 — NDVI / NDWI / NBR (시나리오①·③)

`lib/server/spectralIndex.ts`. STAC 검색 → 결정론적 장면 선택 → COG 창 읽기 → 지수·분류·면적.

### 3.1 파이프라인
```
STAC 검색(과거 60일, 구름<max) → 커버리지 최대·구름 최소 장면(래스터 I/O 없는 순수 기하)
  → geotiff.js Range로 두 밴드의 AOI 창만 → AOI 기준 공통 격자로 리샘플(밴드 해상도 무관)
  → 지수 = (a-b)/(a+b) → 분류·면적(픽셀수 × affine 미터²)
```
- **NDVI** = (NIR B08 − Red B04)/(…) 10m, **NDWI** = (Green B03 − NIR B08)/(…) 10m, **NBR** = (NIR08 B8A − SWIR2 B12)/(…) **20m**
- NBR은 반드시 20m 밴드쌍 — 10m nir를 쓰면 격자가 어긋난다.
- **결정론적 장면 선택**: STAC footprint∩AOI 커버리지로 정렬 → 같은 AOI면 항상 같은 장면(재현성). 구름 없으면 30%→80% 재시도.

### 3.2 폴리곤 클리핑 (시나리오① 핵심)
`place`를 주면 서버가 **Nominatim `polygon_geojson`으로 실제 경계**를 받아 폴리곤 안쪽만 분석한다. 폴리곤은 정점 수백 개라 URL로 못 넘기므로 서버가 지오코딩부터 클리핑까지 처리하고, 픽셀마다 역투영하면 느리니 **폴리곤을 UTM으로 한 번만 투영**해 평면에서 광선교차법으로 판정한다.

**실측(Central Park, 실제 3.41 km²)**: 사각형 9.09 km²(맨해튼 거리 포함, 초고밀도 55.4%) → **폴리곤 3.09 km²(초고밀도 90.2%)**. PNG로 공원이 짙은 녹색 직사각형, 내부 재클린 케네디 저수지까지 정확.

### 3.3 시각화
`/api/spectral/image`가 컬러맵 PNG(ndvi=rdylgn, ndwi=blues, nbr=spectral)를 만들고, 무효 픽셀은 투명이라 배경 지도가 비친다. MapLibre `image` 소스로 bbox 정합. **TiTiler 불필요** — AOI가 작아 PNG 한 장이면 충분(범용 타일링만 롤백됐던 문제와 무관).

## 4. 두 시점 변화 탐지 (시나리오③)

`lib/server/changeDetection.ts`. AWS Tier 1(분광지수 가중 합성) + **dNBR**.

### 4.1 두 지표
- **합성 변화** = Σw|Δ지수|/Σw (NDVI .35, NDWI .15, NBR .15; BSI 미포함). "무엇이든 바뀐 정도".
- **dNBR** = NBR(전) − NBR(후), 양수 = 연소. USGS 심각도 구간.

> ⚠️ **절대 NBR로 연소를 판정하는 건 약하다** — 물·시가지·나지도 NBR이 음수다(강릉에서 2.3% 오탐). 산불 피해는 **화재 전후 차이 dNBR**로 봐야 한다. AWS가 이벤트 전/후 날짜를 브래킷하도록 강제하는 이유.

### 4.2 Tier 2(iMAD)를 뺀 이유
AWS는 iMAD(SVD-CCA 반복 재가중)를 Tier 2로 쓰지만, **Node에 선형대수 라이브러리가 없고 비용 대비 효용이 낮아 의도적으로 생략**했다(문서 권고와 별개 판단).

### 4.3 실측 검증
```
Dixie Fire(북캘리포니아) 2021-07-08 → 09-16   dNBR +0.615, 고심각도 46.8%, 연소 64.4km²
Pacific Palisades 2025-01-02 → 01-17          dNBR +0.599, 고심각도 40.2%, 연소 1.74km²
```
PNG 육안: 고심각도 적색이 뚜렷. `/api/change/image?layer=composite|dnbr`.

> **울진 산불(2022-03) 검증 실패는 구현이 아니라 데이터** — 겨울 산악 장면이 지형그늘·구름으로 유효 픽셀 2%만 남았다. 여름 화재(Dixie)로 재검증해 구현이 옳음을 확인.

## 5. 지수 2시점 비교 (시나리오② 수자원)

`compareIndex()`. 변화탐지가 "무엇이든 바뀐 정도"라면 이건 **"특정 지수 면적이 얼마나 늘고 줄었나"**. 같은 AOI를 두 시점으로 돌려 클래스별 면적 증감을 낸다(목표일 최근접 장면 + 30일 창).

> ⚠️ **NDWI 임계값이 물을 못 잡는다** → SCL로 대체. L2A가 어두운 수면 반사율을 음수로 과보정해 NDWI가 −1로 뒤집힌다(폴섬호 수체 0.02km², 실제 만수위 46km²). PNG로 보면 호수 Y자 형태는 또렷해 상대신호는 맞고 절대값만 어긋난 것. **수체 면적은 SCL(장면분류) water 클래스**로 산출한다.

**실측(폴섬호, 같은 9월)**: SCL 수체 **25.42 km²(2021 가뭄) → 36.64 km²(2022 회복), +44.2%**. 만수위 대비 현실적.

## 6. 광역 토지변화 스캔 (시나리오④) — Clay 임베딩

`lib/server/regionChange.ts`. 픽셀 변화탐지의 **상위 계층** — 1.28km 셀로 주/국가 규모를 훑는다.

### 6.1 방식
LGND 공개 **Clay v1.5 임베딩**(256차원, GeoParquet)의 두 시점 **같은 cell_id를 조인**해 코사인 유사도를 잰다. 낮을수록 변화가 크다. geohash-2 파티션으로 관심 지역만 로드.

### 6.2 적응형 캘리브레이션 (핵심 판단)
AWS 고정상수(SIM_HIGH 0.90)는 콜로라도 **야생지** 기준이라 도시가 많은 한국에선 안정지역도 변화로 과대보고한다(서울 코사인 중앙값 **0.794** → 고정값이면 32%가 "변화"). → **지역 자체 분포의 p10~중앙값으로 정규화**해 17.5%로 선별. 문서가 예고한 지역 의존성을 실제로 해결.

### 6.3 실측 검증 (API 직접)
```
서울권 2019-07 vs 2024-07: 조인 1303셀, 코사인중앙값 0.794, 적응임계 0.466,
                           변화 228셀, 변화면적 373.6km²
핫스팟 @ 남양주·다산·검단신도시 ← 2019~2024 실제 개발 전선에 정확히 적중
```
- **파티션 240MB → LRU 캐시(2개)**: 재조회 55초 → **0.04초**.
- **parquet-wasm WASM 사이드카**가 번들에서 누락(ENOENT) → `next.config` `serverExternalPackages`로 런타임 로드.

> ⚠️ **미검증**: 채팅 E2E·지도 오버레이. dev 서버가 stale agent 청크를 서빙(spectral은 되는데 신규 scan은 안 됨)해 `.next` 삭제·캐시삭제·재시작으로도 못 뚫었다. 소스·API·타입은 정상 — 프로덕션 빌드(content-hash 청크)나 하드 리프레시로 반영될 것.

## 7. LLM 구성 고도화

AWS 문서의 LLM 운영 기법을 우리 Ollama 스택에 이식.

### 7.1 num_ctx 절단 버그 (가장 중요)
**Ollama는 `num_ctx` 미지정 시 프롬프트를 조용히 앞에서 잘라낸다.** qwen3:8b는 40,960 지원인데 실제 2,050에서 절단됐다(마커 회수 실증: 절단 시 "42" 환각, 지정 시 정답). `[근거]`가 통째로 사라져 그라운딩이 무력화되고 있었다. `lib/server/llm.ts` 단일 창구가 num_ctx(텍스트 16384/VLM 8192)·온도(0.1)를 강제하고 `prompt_eval_count`를 노출한다.

### 7.2 Qwen 자연어 서술 (측정 후 도입)
"수치를 서술만" 하는 작업의 왜곡 위험을 실측(temp 0.1 + num_ctx + "수치 변경 금지" → 5회 반복 지어낸 수치 0건). 도입하되 3중 안전장치: **원본 수치 블록 병기**(라벨 짝 오류 검증), **품질 경고(⚠️) 붙으면 서술 생략**(신뢰 못 할 수치를 매끄럽게 포장 방지), 실패 시 조용히 수치만.

### 7.3 지오코딩 교차검증
AWS `get_best_geometry` 이식 — 후보 5개 중 **면적 과도(국가/주) 기각 + 가장 큰 것 채택**(Nominatim 1순위는 역·건물 POI가 많다: "울진" → 울진역 1km²가 아니라 울진군 174km²). France 438만km² 기각 → 폴백. 오매칭이 조용히 엉뚱한 지역을 분석하는 것을 코드 레벨에서 차단.

## 8. 에이전트 배선 + 라우팅

`resolveIntent`(정규식 결정론)에 신규 도구 배선. 응답은 결정론 템플릿으로 조립(수치 왜곡 0) + 선택적 Qwen 서술.

| 도구 | 트리거 | 결과 |
|---|---|---|
| `spectral_index` | 식생/수체/연소 + 장소 | NDVI/NDWI/NBR 통계 + PNG |
| `change_detection` | 변화/산불피해 + 날짜 2개 | 합성+dNBR + PNG |
| `compare_index` | NDVI/NDWI + 날짜 2개 | 지수 면적 증감 |
| `scan_region_change` | 광역/스캔 + 연도 2개 | Clay 변화 핫스팟 |

**라우팅 충돌 해소**: 날짜 2개 질의에서 **NDVI/NDWI 명시 → compare_index**, **NBR·산불·변화 → change_detection**(dNBR이 전후 NBR 차이라 산불엔 이게 맞음). **findCity 부분문자열 버그** 수정("Folsom **La**ke"의 'la'가 LA로 매칭돼 971km² 오분석) — ASCII 키에 단어 경계 적용.

## 9. 함정·교훈 모음 (AWS 원본 대비 교정 포함)

- **음수 반사율 처리 3단 진화**: ①방치=분모 0교차로 지수 폭발(-1197) → ②전부 NaN=울창한 수관 삭제(센트럴파크 투명, 서울 초고밀도 55.6→20.6%) → ③**0 클램프 + 밝기 하한(MIN_BRIGHT)**. "음수냐"가 아니라 "픽셀 전체가 어두운가"가 기준.
- **스케일/오프셋**: L2A baseline 04.00+ 는 refl=DN×0.0001−0.1. 오프셋은 덧셈이라 (a-b)/(a+b) 분모가 상쇄 안 됨 → 원시 DN 계산은 편향(AWS `ndvi_utils.py:76` 버그).
- **격자 정합**: 10m(NDVI)와 20m(NBR)가 512×512 vs 443×512로 어긋남 → AOI 기준 공통 격자 강제.
- **장면이 계절을 섞음**: 구름 우선 정렬이라 "사건 직후"를 요청해도 몇 주 전 장면 선택 → 목표일 최근접 우선.
- **SCL 마스킹의 딜레마**: 구름·눈은 버려야 하지만 class 2(지형그늘)까지 버리면 겨울 산악 유효면적 붕괴(0.08km²) → class 2 보존.
- **NDWI 임계값 ≠ 수체**: SCL water가 정석.
- **Clay 캘리브레이션은 지역 의존**: 적응형 정규화.
- **Ollama num_ctx**: 미지정 시 조용한 절단.
- **parquet-wasm**: WASM 사이드카 번들 누락 → serverExternalPackages.
- **dev 청크 캐싱**: 스테일 청크가 신규 라우팅을 안 반영(프로덕션 빌드는 content-hash라 무관).
